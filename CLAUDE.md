## Project state

Phase 1 (Foundation) is complete: navigation, file operations, multi-select, drag-and-drop, breadcrumbs, and the "This PC" landing view all work. A sidebar folder tree was built and then deliberately removed at the user's request — don't re-add it without being asked. `EntryTable`'s per-row context menu has Open / Open with (files only, not folders — see below) / Rename / Copy / Cut / Delete / Properties; right-clicking empty space (not a row) shows a folder-level background menu (New Folder / New File / Paste / Add-or-Remove Favorite / Refresh) instead. Right-click still does nothing on Sidebar/This PC items (native context menu suppressed app-wide there too, no custom menu built for them — no clear use case yet; don't invent actions speculatively).

**Copy and Move are both clipboard-based, not "type a destination path" modals** — non-technical users don't think in filesystem paths, and typing one was reported as confusing. `copySelected()`/`cutSelected()` call `store.setClipboard(paths, "copy" | "cut")` (`file-explorer.store.ts`'s `clipboard: ClipboardState | null`, not persisted) instead of opening `PromptModal`; pasting happens via the context menu's Paste item (`useFileExplorer.ts`'s `pasteIntoCurrent()`), which copies/moves every clipboard path into whatever folder is currently browsed and, for a cut, clears the clipboard afterward (a copy's clipboard survives a paste, so it can be pasted again elsewhere — matches Explorer). This is the one pair of actions in the whole app that intentionally mirrors OS Explorer's own cut/copy/paste model rather than inventing a Schlag-specific flow, precisely because it's the interaction non-technical users already know. `Move`'s old destination-path `PromptModal` case was removed outright (not left dead) once Cut shipped — `PromptKind` is now just `"new-folder" | "new-file" | "rename"`, and `validateFolderPath`/the suggestions-based Combobox path in `promptConfig.ts` went with it (the New Folder/New File/Rename prompts never needed a folder path field to begin with).

`ContextMenuState` gained a `background: boolean` field so `ContextMenu.tsx` can render the reduced folder-level variant; `EntryTable.tsx`'s wrapping `<div>` (both the populated and empty-folder states) gained an `onContextMenu` handler mirroring the existing `onClick`-to-clear-selection pattern (`e.target === e.currentTarget`, so a row's own context menu doesn't also bubble into triggering the background one) — wired only for the plain directory listing, not search results or "This PC", since "which real folder do these actions target" isn't well-defined for either. The background menu's New Folder/New File/Refresh/Favorite-toggle items call the exact same `useFileExplorer.ts` functions the `Toolbar` buttons already use (`newFolder`, `newFile`, `store.refresh`, and a small `toggleCurrentFavorite()` wrapper around `store.toggleFavorite(store.currentPath)`) rather than duplicating that logic — the context menu is just a second entry point into behavior that already existed.

**"Open with…" is hidden entirely for folders**, not just disabled — Windows' own `ShellExecuteExW "openas"` verb is a file-association concept that doesn't meaningfully apply to directories, and showing a dead-looking disabled button for every folder was worse than not showing it. `ContextMenu.tsx` takes a `selectedIsDir` prop (`useFileExplorer.ts`: `selectedEntries.length === 1 && selectedEntries[0].is_dir`) and conditionally skips rendering the button rather than passing `disabled`.

**A cut entry is visually dimmed (`opacity-50` on its `<tr>`) until it's pasted or the cut is replaced**, matching Explorer's own "in transit" treatment — otherwise there's no visual difference between an item that's about to move and one that isn't, and a user can lose track of what they cut. `EntryTable`'s `cutPaths` prop (`useFileExplorer.ts`'s `cutPaths = store.clipboard?.op === "cut" ? store.clipboard.paths : []`) is checked per-row in `EntryRow`; wired into both the plain directory listing and the search-results `EntryTable`, since a cut item can legitimately show up in either.

**Delete's confirmation is a real modal (`ConfirmModal.tsx`), not `window.confirm`** — the last native browser dialog left in the app. A separate component from `PromptModal` rather than a variant of it, since a yes/no confirmation has no text field and no validator; its Confirm button uses the `bg-error-container`/`text-on-error-container` pairing already established for the Delete row's hover state in `ContextMenu.tsx`, applying that same destructive-action treatment to a full button instead of inventing a new red. `deleteSelected()` now just opens it (`store.deleteConfirmOpen`, mirroring the `activePrompt`/`clipboard` nullable-state pattern); the actual `deleteEntry` calls moved into a new `confirmDelete()`, run only once the user clicks through.

**`create_dir`/`create_file`/`rename_entry` (`fs_ops.rs`) all reject a name collision up front instead of silently corrupting or erroring cryptically.** Three real, reported bugs, same root cause: Rust's `fs::write`/`fs::rename` have no "fail if the destination exists" mode. `create_file` used `fs::write(path, [])`, which silently truncated an existing file with that name to empty (no error, no visible new entry, which read to the user as "nothing happens" but was actually quiet data loss); naming it the same as an existing *folder* surfaced Windows' raw `"Access is denied. (os error 5)"` instead, since `CreateFileW` denies write access to a directory rather than reporting a collision. `rename_entry` was worse — Rust's Windows implementation of `fs::rename` always calls `MoveFileExW` with `MOVEFILE_REPLACE_EXISTING`, so renaming a file onto an existing name silently **deleted that other file outright**, no error, no trace, confirmed live by a real rename that made an existing file "just disappear." All three commands now `Path::new(&path).exists()`-check before touching disk and return one friendly `"A file or folder with that name already exists."` regardless of what's already there — covered by `fs_ops.rs` tests asserting the existing file's content survives untouched (for both create-collision cases and the rename-collision case) and that file-vs-file, file-vs-folder, and rename-vs-existing collisions all produce the same clean error (surfaced inline by `PromptModal`, same as any other validation failure). `copy_entry`/`move_entry` don't need this guard — they already run every destination through `unique_destination()` to auto-number instead of colliding (see below); `rename_entry` deliberately does *not* get that same auto-numbering treatment, since silently renaming to a different name than what the user typed, without telling them, would be its own kind of confusing.

**Cutting a file and pasting it back into the same folder it's already in is a no-op, not a numbered duplicate.** `pasteIntoCurrent()` used to call `moveEntry` unconditionally, which — same as any other destination collision — went through the backend's `unique_destination()` and produced a `" (1)"`-suffixed copy instead of actually moving anything, which is correct for a *copy* pasted into its own folder (Explorer does the same: Ctrl+C, Ctrl+V in place makes a duplicate) but wrong for a *cut*, where there's nowhere to move to. `pasteIntoCurrent()` now filters `clip.paths` to drop any path whose `dirname()` already equals `store.currentPath`, but only when `clip.op === "cut"` — a copy pasted in place still goes through and still numbers, unchanged. If filtering leaves nothing to move, the function returns immediately without calling the backend, refreshing, or clearing the clipboard, so the cut is still live and pasteable somewhere else afterward.

- `npm run build` — `tsc` then `vite build`.

- `cargo test` (from `src-tauri/`) — Rust unit tests in `fs_ops.rs` (quick access dirs, drive enumeration, recursive copy, move), `database.rs` (upsert/update/delete against a temp SQLite file), `indexer.rs` (scan against a temp directory tree, notify-event application, excluded-directory pruning for both the scan and the live watcher), and `search.rs` (substring/escaping, each filter individually and combined, result cap — all against a temp SQLite file seeded through the same `database::open()`/`upsert_batch()` path production code uses, so the FTS5 triggers are exercised for real, not mocked).

## Architecture

Tauri app: React/TypeScript frontend calls Rust commands via `invoke()`; the frontend never touches the filesystem directly. All filesystem work lives in `src-tauri/src/fs_ops.rs` (single module — `home_dir`, `quick_access_dirs`, `list_drives`, `list_dir`, `create_dir`, `create_file`, `rename_entry`, `delete_entry` (via the `trash` crate, not `fs::remove_*`), `copy_entry`/`copy_dir_all`, `move_entry` (renames, falls back to copy+delete across volumes), `open_with_dialog`, `show_properties`). `dirs` crate resolves platform folders; drives are enumerated by probing `A:`–`Z:` (Windows-specific, not yet cross-platform).

`open_with_dialog`/`show_properties` invoke the real native Windows dialogs via `ShellExecuteExW` (`windows` crate, Windows-only dependency in Cargo.toml) with the `"openas"`/`"properties"` verbs and `SEE_MASK_INVOKEIDLIST` (required for `"properties"` to work), instead of reimplementing an app-picker or a General/Security/Sharing panel — see plan.md's File Operations section. This is the actual public Win32 API Explorer itself uses. Two informal alternatives were tried first and both silently no-op'd (exit 0, no output, no visible window) depending on the calling process's context: `rundll32 shell32.dll,OpenAs_RunDLL` (an undocumented internal DLL export, not a supported API) and PowerShell + `Shell.Application` COM `InvokeVerb('properties')` (COM apartment state sensitive — `InvokeVerb` needs STA, PowerShell defaults to MTA). Don't go back to either without a concrete reason.

**Indexing (`database.rs` + `indexer.rs`)**: on app startup (`lib.rs`'s `setup()` hook), one `std::thread` is spawned that owns a SQLite connection (`rusqlite`, bundled), shared as `Arc<Mutex<Connection>>` between the scan and the live-event drain thread (see below) — nothing else writes to the DB, so this Mutex is the only writer-side synchronization needed. `indexer::spawn()` scans the user's home directory first (via `fs_ops::home_dir()`), then every drive — so useful results land quickly instead of waiting for a full drive to finish; the home dir gets walked twice total (once standalone, once again inside its drive's full walk) since nothing tracks already-covered paths, an accepted tradeoff since upserts are idempotent. Each `scan_drive()` call lists the root's immediate children on the calling thread, then walks each child subdirectory's subtree **in parallel via `rayon`** (`par_iter().for_each`), with the shared `Mutex<Connection>` so parallel workers serialize on the (fast, batched) writes. `EXCLUDED_DIR_NAMES` (`node_modules`, `.git`, `.cache`, `$RECYCLE.BIN`, `System Volume Information`, plus package-manager/dependency caches — `.cargo`, `.rustup`, `.npm`, `.nuget`, `.gradle`, `.m2`, `.venv`, `venv`, `__pycache__`, `site-packages` — and build output — `target`, `dist`, `build`, `.next`) is checked via `walkdir`'s `filter_entry` so those trees are pruned entirely — never even listed — rather than walked and discarded; it's a hardcoded list, not a user setting (add a Settings toggle in Phase 6 if that's ever needed). The `notify` watcher doesn't get this pruning for free — it recursively watches whole drives regardless of what the initial scan skipped — so `apply_event` separately checks every path component of an incoming event against the same list (`path_is_excluded`) before touching the DB; without it, live filesystem activity inside an excluded folder (e.g. a dev server writing into `node_modules/.vite`) would silently re-populate it. Confirmed live: a real scan initially showed 2,000+ false-positive `node_modules` hits that turned out to be filenames merely *containing* that substring (bundler chunk names, `.asar` bundles) — the exclusion itself only ever matches an exact path *component*, not a substring. **This check is per-path, not per-event** — `apply_event` used to skip a whole multi-path event if *any* of its paths were excluded, which broke deletion: deleting a file (via Explorer or `fs_ops::delete_entry`'s `trash` crate) moves it into `$RECYCLE.BIN` rather than removing it outright, which `notify` reports as a rename (old real path → new `$RECYCLE.BIN` path) — and since the new path is excluded, the *whole event* got skipped, so the old path never got deleted from the index and the file kept showing up in search forever. Verified live: sent a real file to the Recycle Bin via `Shell.Application`'s `InvokeVerb("delete")` COM call (the actual mechanism Explorer uses, not a plain `rm`) and confirmed it dropped out of search within seconds after the fix.

`start_watchers()` registers a recursive `notify` watcher per drive **before** the initial scan starts (not after), and `drain_events()` (applying create/modify/remove/rename events) runs on its **own thread concurrently with the scan**, not sequentially after it. Both of these were fixed after a real, reachable bug: registering watchers only after the full scan meant a file created *during* a long scan fell into a permanent gap (the watcher didn't exist yet when the OS-level create event fired) and never got indexed until the next full app restart happened to re-cover it. Draining sequentially-after-scan (an earlier, partial fix) turned that into "doesn't show up until the *entire* scan finishes," which for a real multi-million-file `C:\` can be minutes — still indistinguishable from broken to a user who just added a file. Verified live: created a file mid-scan (confirmed via log — "initial scan complete" hadn't printed yet) and it was searchable within ~3 seconds via the concurrent drain thread. Deliberately **not** `tokio`-based: rusqlite/walkdir/notify are all synchronous, and nothing else in this codebase is async — bare OS threads (plus `rayon` for the one CPU/IO-bound parallel step) are simpler and consistent with `fs_ops.rs`'s style. Progress is exposed via `IndexStatus`, in-memory atomics (not a live `SELECT COUNT(*)`) polled by the `index_status` command and the frontend's `useIndexStatus()` hook. DB lives at `{app_data_dir}/index.db`. Walkdir/notify errors (permission-denied, e.g. leftover access-denied subfolders) are logged via `tracing::warn!` and skipped, never propagated.

**Watching a whole drive via `ReadDirectoryChangesW` (what `notify` uses on Windows) can silently drop events under load** — confirmed live: deleting a file via the `trash` crate produced *no* notify event at all despite the file genuinely being gone, while the same delete via `Shell.Application`'s `InvokeVerb("delete")` COM call did produce one. This isn't a bug in this codebase's code, it's an inherent limitation of recursively watching an entire drive with a lot of background churn — and this app's own `index.db`/`index.db-wal` writes were adding to that churn unnecessarily (every DB write is itself a filesystem change the recursive watcher picks up and re-indexes as a normal file, a self-referential loop), so `com.carlo.schlag` (this app's own data directory name) is on `EXCLUDED_DIR_NAMES` now specifically to cut that noise. Since a missed event can't be fixed at the `notify` layer, `prune_stale_entries()` self-heals instead: once at startup (before the scan, off the `Arc<Mutex<Connection>>` shared with everything else), it stat-checks every already-indexed path in parallel via `rayon` and deletes rows for files that no longer exist. This is a real, if imperfect, mitigation — a deleted file can still show up in search for as long as the app keeps running after a missed event, but it will always be gone again after the next restart, instead of the row persisting forever. A full continuous reconciliation (polling this while the app runs, not just at startup) wasn't built — the startup pass was judged sufficient for now; revisit if missed events turn out to be common in practice.

**Search (`search.rs`)**: `search_files` is a Tauri command that queries the same `index.db` the Phase 2 indexer writes to, but through its **own** `Connection` — WAL mode (already on) explicitly supports concurrent readers alongside one writer, so this is the minimal-diff way to add the first reader rather than routing queries through the indexer's writer thread. Opened once in `lib.rs`'s `setup()`, wrapped in `Mutex<Connection>` (needed since `Connection` isn't `Sync`), `.manage()`d as Tauri state. Filename substring matching goes through an FTS5 virtual table (`files_fts`, `tokenize='trigram'`, external-content on `files`/`id`) rather than `files.name LIKE '%x%'` directly — a plain B-tree index can't accelerate a leading-wildcard `LIKE`, and a real substring-plus-filter query against the live ~1.5M-row index measured 4–7+ seconds before this, which is a real, felt slowness, not a theoretical one. `files_fts` is kept in sync via `AFTER INSERT/UPDATE/DELETE` triggers on `files` (created in `database::open()`, cheap regardless of table size), so ongoing indexing (including Phase 2's live `notify`-driven updates) keeps it current for free. Extension/size/date/folder/regex filters are plain `AND` clauses against `files.*` in the same query, dynamically appended only for filters that are actually set. Query strings under 3 characters fall back to a plain `files.name LIKE` scan — the trigram tokenizer can't extract a trigram from anything shorter, and such broad queries hit the result cap fast enough anyway (measured ~1ms) that the fallback's full-scan cost doesn't matter in practice.

**Filename search has a second matching mode, "keywords" (`build_keyword_match`), alongside the default "phrase" mode (`build_phrase_match`) above** — both are branches of `build_query(query, filters, keyword_mode)`, selected by a `keyword_mode: bool` parameter threaded through `run_query`/`search_files` from the frontend. Phrase mode (the pre-existing behavior) requires the whole query to appear as one contiguous run of characters; keyword mode requires every whitespace-separated word to appear *somewhere* in the name, in any order, but rejects a name containing only some of the words — explicit user request ("scatter the search value, but the items SHOULD contain those keywords... do not show a file if it contains like tester 123 if a file contains only tester man test"). `build_keyword_match` partitions the query's words by the same 3-character trigram threshold `build_phrase_match` already used: words ≥3 chars each become their own quoted `files_fts MATCH` phrase (all ANDed together into one combined MATCH string, since FTS5 only allows one `MATCH` per table per query), words <3 chars each become their own `files.name LIKE '%word%' ESCAPE '\'` clause ANDed in alongside — covered by `keyword_mode_handles_a_mix_of_short_and_long_words`, which mixes a 2-char and a 6+-char word in the same query. Filter-appending (extension/size/date/folder/regex) is unchanged and shared by both builders.

**The `folder` filter matches a trailing path separator, not just a raw prefix** (`path LIKE '{folder}\%'`, not `path LIKE '{folder}%'`). A plain prefix match also matches the folder's own row — any string trivially "starts with itself" — which surfaced as a real, confusing bug once folder-scoped search (below) shipped: search from inside a folder for a query matching that folder's own name (e.g. standing in `Documents`, searching "Documents") returned the folder itself as a "result," since its `name` matched the query *and* its `path` trivially satisfied the scope filter. The fix appends a separator before the wildcard (escaped correctly for `ESCAPE '\'` — the whole `folder + "\\"` string goes through `escape_like` together, not the folder alone with a raw `\` tacked on after, which would misparse as an escape sequence for the following `%`) so the filter only matches *contents* of the folder, never the folder entry itself.

**Content search (`content_index.rs`)**: a second, independent index — a Tantivy full-text index at `{app_data_dir}/content_index/`, separate from the SQLite `files`/`files_fts` filename index above — over what's *inside* extractable files, checked case-insensitively via `is_extractable()` against `EXTRACTABLE_EXTENSIONS`: `.txt`/`.md`/`.markdown`, `.pdf`/`.docx`/`.xlsx`/`.pptx`, and a broad set of code/config extensions (`.js`/`.ts`/`.py`/`.rs`/`.go`/`.java`/`.c`/`.cpp`/`.cs`/`.rb`/`.php`/`.sh`/`.ps1`/`.sql`/`.html`/`.css`/`.json`/`.yaml`/`.toml`/`.xml`/`.vue`/`.svelte`, etc.) — added on request once it was clear these are just plain text, no different from `.txt`/`.md`, so `extract_text()`'s dispatch match has a single `_ => extract_plain_text(path)` fallback arm covering all of them rather than one match arm per extension. **Legacy binary Office formats (`.doc`/`.xls`/`.ppt`) are a deliberate exclusion, not an oversight** — they're the old OLE2/Compound File Binary container, structurally nothing like the "zip of XML parts" the DOCX/XLSX/PPTX extraction below depends on; supporting them would need a real binary-format parser, not a longer whitelist, and they're rare in practice since Office defaulted to the modern formats back in 2007. `extract_text()` dispatches by extension: `String::from_utf8_lossy` over raw bytes for the plain-text fallback arm (never `read_to_string`, which would hard-fail and skip an otherwise-fine file over one stray non-UTF8 byte); `pdf-extract` for PDF; `calamine` for XLSX (iterates every sheet's used cells); and one shared zip+XML helper for **both** DOCX and PPTX, since both formats are "a zip of XML parts" (OOXML) and both put visible text runs in an XML element with local name `t` regardless of namespace prefix (`w:t` in DOCX, `a:t` in PPTX) — this was a deliberate choice over higher-level crates like `docx-rs` or the several surveyed pptx-specific crates (`pptx-to-md`, `rust-pptx`, `msoffice_pptx`), which are lower-maturity/smaller-community than `zip`+`quick-xml` (which most of them use internally anyway) — one low-level dependency pair covers two formats instead of two higher-level ones. PPTX slides are read via `ppt/slides/slideN.xml`, sorted **numerically** by N (`slide_number()`), not lexicographically — a plain string sort would put `slide10.xml` before `slide2.xml`. A source file over ~20MB is skipped outright, and extracted text over ~500,000 chars is truncated, not rejected — both are `ponytail:`-documented ceilings bounding pathological-file hang risk and per-doc Tantivy storage growth.

**Content indexing runs on its own dedicated thread/channel (`indexer.rs`'s `ContentEvent`/`run_content_indexer`), independent of the metadata scan**, because content extraction (parsing a PDF/DOCX/etc.) is far slower per-file than the metadata scan's stat calls — inlining it would slow down the already-fast filename index. `scan_drive`/`walk_subtree`/`apply_event` all additionally queue a `ContentEvent::Index(path, modified_ms)`/`Remove(path)` (via `queue_content_event`/`queue_content_removal`) alongside their existing SQLite work, for any extractable path. One dedicated consumer thread owns the Tantivy `IndexWriter` exclusively and drains that channel — no `Mutex<IndexWriter>` needed (unlike `Mutex<Connection>`), since a single-consumer-via-channel is the natural fit for a resource that isn't safely shared across threads for concurrent writes anyway. Commits are batched (every 50 docs or 5 seconds, whichever comes first), same batching rationale as the metadata scan's `flush_batch`.

**Re-extraction is skipped when a file's content hasn't changed, via a new `content_index_state(path, indexed_mtime)` SQLite table** (`database.rs`, created the same `CREATE TABLE IF NOT EXISTS` way as everything else — no `ALTER TABLE` migration needed). Phase 2's metadata scan already re-walks every drive on **every app launch** (upserts are idempotent, so this was always fine for cheap metadata) — but content extraction is expensive enough that redoing it unconditionally on every launch would be a real, felt slowdown for anyone with a real Documents folder full of PDFs/Office docs. Before extracting, `run_content_indexer` checks this table; a path is only re-extracted if its current `modified_ms` no longer matches what's recorded. This also means **no separate backfill step was needed** for the Tantivy index's first-ever build: the table starts empty, so the very next full scan naturally treats every extractable file as new — exactly the backfill behavior wanted, for free, mirroring how `files_fts`'s own triggers need no backfill on a brand new install.

**`search_content` exposes snippets as plain `fragment` text + `highlight_ranges`, never pre-built HTML.** Tantivy's own `Snippet::to_html()` wraps matched terms in `<b>` but does **not** HTML-escape the surrounding fragment — and that fragment is raw *file content* the app has no control over, so rendering it as trusted HTML would be a real XSS vector (a file whose text happens to contain `<script>`). The frontend builds highlighted spans itself via plain (auto-escaping) JSX string-splitting instead (`lib/highlightSnippet.ts`'s `splitHighlights`) — `dangerouslySetInnerHTML` is not used anywhere in this feature.

**Tantivy's `Snippet::highlighted()` ranges are *byte* offsets into the UTF-8 fragment — but the frontend is JavaScript, which indexes strings in UTF-16 code units.** These coincide for ASCII text but diverge for anything else (accented characters, non-Latin scripts, emoji), which would silently mis-highlight or crash on a non-character-boundary slice. `content_index.rs`'s `byte_ranges_to_utf16()` converts before the result ever reaches the frontend (building a byte→UTF-16 offset map via `char_indices()`), so the frontend can just slice its string directly with no encoding awareness of its own — covered by a unit test using a multibyte (`é`) fixture, not just ASCII.

**Folder scoping for content search is a post-filter over an overfetched candidate window, not a native Tantivy query** — `search_content` asks Tantivy for `RESULT_LIMIT * OVERFETCH_FACTOR` (4×) candidates when a folder filter is set, then filters by path prefix in Rust before truncating to the real limit. `ponytail:`-documented ceiling: if a folder-scoped query needs more than that overfetch window to find enough in-folder matches, results may be incomplete; revisit with a real Tantivy-side path-prefix query if that proves common in practice. Content search's own metadata (`name`/`is_dir`/`size`/`modified_ms`) is looked up from the existing SQLite `files` table per surviving candidate, not duplicated into Tantivy's schema — Tantivy only stores `path`/`content`/`modified_ms`, the latter unused by search itself (the `content_index_state` table above is the actual source of truth for the re-extraction-skip decision).

**Content search's own core query logic lives in a plain `run_content_query()` function, separate from the `#[tauri::command] search_content` wrapper** — mirrors `search.rs`'s own `build_query`/`run_query` split from its `search_files` command, and for the same reason: the actual logic (query parsing, folder-prefix filtering, SQLite metadata join, snippet generation) is directly unit-testable without needing to construct Tauri `State` wrappers.

**Content search wraps every query in a quoted phrase (`phrase_query()`) before handing it to Tantivy's `QueryParser`, rather than parsing the raw string.** Tantivy's default parser splits unquoted input on whitespace and scores each word independently (any-of-these-terms), so an unquoted search for "am a file" would match any document containing "am", "a", or "file" *anywhere*, not that run of words together — a real, reported UX bug ("the search value shouldn't break") once content search was exposed to a general search box rather than a narrow "find this exact term" field. Filename search never had this problem: its own FTS5 phrase (`search.rs`'s `fts_phrase()`) already wraps the whole query as one quoted MATCH, since the trigram tokenizer's substring-search design made that the natural choice from the start. Wrapping unconditionally (not just for multi-word input) is deliberate — a single-word query wrapped in quotes parses identically to the same word unquoted, and this is meant to behave like "find this text," not expose a query language with AND/OR/field:value syntax. Covered by a test that indexes one document containing the phrase verbatim and a second containing the same three words scattered out of order, asserting only the first matches.

**Content search gets the same phrase-vs-keywords split as filename search, via `keyword_query()` next to the existing `phrase_query()`.** `run_content_query`/`search_content` take the same `keyword_mode: bool` parameter as `search.rs`'s equivalents, and build the query text passed to Tantivy's `QueryParser` accordingly: `phrase_query(query)` (unchanged) for the default, or `keyword_query(query)` for keyword mode. `keyword_query` deliberately does **not** use Tantivy's `set_conjunction_by_default()` API — it quotes each whitespace-separated word individually via the existing `phrase_query()` helper and joins them with a literal `" AND "` in the query text itself, so a raw word containing something that looks like Tantivy query syntax (a colon, a paren) still can't reach the parser unescaped, same reasoning already documented above for `phrase_query`. Covered by an extension to the existing phrase-mode test (`run_content_query_treats_the_query_as_a_contiguous_phrase_not_separate_words`): the same two fixture documents (one containing "am a file" verbatim, one containing the same three words scattered) both match in keyword mode, while a query substituting one word for one that's absent from both matches neither.

**Preview (`preview.rs`)**: deliberately the smallest possible new backend surface, because almost every format category needed **zero** new Rust logic. `app.security.assetProtocol` (`tauri.conf.json`, `{ "enable": true, "scope": ["**"] }` — broad scope for the same reason `opener:allow-open-path` already uses `**`: files worth previewing can be on any drive, not some app-scoped folder) is what lets the frontend hand the webview a direct file URL (`convertFileSrc()`) for Images/Video/PDF — the webview decodes/renders these natively (`<img>`, `<video controls>`, and `<embed type="application/pdf">`, the last relying on WebView2's own built-in Chromium PDF viewer for real paginated rendering, not a bundled library). **Enabling `assetProtocol` requires a full `npm run tauri dev` restart**, same as any other `tauri.conf.json` change — Tauri compiles capability/config into the binary at build time. Markdown/Text/Office preview reuses `content_index::extract_text()` outright — that function was already `pub` and already had its own per-type dispatch and size ceilings from Phase 3's content-search work, so `preview_text` is a one-line wrapper with no extraction logic of its own; it's the frontend's `previewKind()` categorization (`lib/previewKind.ts`) that ensures this command is only ever called for markdown/text/office paths — PDF's own `extract_text` branch (used for content search's indexing, not preview) is simply never reached from here. **Archive preview (ZIP only) is the one genuinely new piece of backend logic** — `read_archive_entries()` (split from the `list_archive_entries` command wrapper, mirroring `search.rs`'s `build_query`/`run_query` split, for the same directly-unit-testable reason) opens the file via `zip::ZipArchive` (already a dependency, from the DOCX/PPTX extraction work) and returns a flat `(Vec<ArchiveEntry>, bool)` — name/size/is_dir per entry plus a `truncated` flag, capped at a `ponytail:`-documented 2000 entries. Non-ZIP archive formats (rar/7z/tar.gz) aren't supported — `zip` only reads ZIP.

**Office preview shows extracted plain text, not the original document layout** — a small banner in `PreviewPane.tsx` says so explicitly ("Showing extracted text, not the original formatting") rather than silently shipping something that looks like it's trying and failing to be a real DOCX/XLSX/PPTX renderer. Real WYSIWYG rendering (fonts, tables, slide layout) would need a much heavier renderer per format; reusing the existing text-extraction pipeline was the deliberate cheapest-correct choice for this pass, upgradeable later if that proves insufficient.

**Markdown rendering uses `react-markdown`, not a markdown-to-HTML-string library plus `dangerouslySetInnerHTML`.** Same XSS reasoning already documented for content-search snippets: the markdown being rendered is arbitrary file content the app doesn't control, so trusting it as raw HTML would be a real vector. `react-markdown` parses to an AST and renders through actual React elements — raw HTML embedded in a markdown file is neither executed nor unescaped-inserted by default (no `rehype-raw` plugin is used), so there's no `dangerouslySetInnerHTML` anywhere in this feature either. Its handful of rendered elements (headings/paragraphs/lists/code/blockquotes/links) are hand-styled via a `.markdown-preview` CSS block in `App.css` rather than pulling in `@tailwindcss/typography` as a dependency — that plugin's generic defaults wouldn't match this app's specific density/type scale, and only a handful of elements ever need styling here.

It also adds a new **"Open file location"** action (`FolderOpen` icon, between Open and Open With) — navigates to the entry's parent folder and selects it there (`useFileExplorer.ts`'s `openFileLocation`), most useful for a search result (which can live anywhere) and why `ContextMenu`'s `onOpenLocation` prop is optional and omitted by the normal directory-listing usage — showing it for every regular right-click would just re-confirm the folder you're already looking at, and real Windows Explorer doesn't show it for ordinary files either, only shortcuts. `ContextMenu`'s own z-index was bumped from `z-50` to `z-[70]` so it renders above `SearchModal`/`PromptModal`/`ConfirmModal` (all `z-[60]`) — it never visibly overlapped any of them before (clicking any of its buttons closes it via the same window-click-listener bubbling that would open a modal), but nesting it inside the search overlay for the first time made the stacking order actually matter.

[store.promptTarget] : selectedEntries` / `deleteEntries = store.deleteTarget ?? selectedEntries` and threads those into `getPromptConfig`/`deleteMessage`/`confirmDelete` — `lib/promptConfig.ts` itself needed **no changes**, since it already just takes whatever `selectedEntries`-shaped array it's given. - `store/search.store.ts` — Zustand, ephemeral (not persisted, unlike `file-explorer.store.ts`): `isOpen`, `query`, `filters`, `scopeToFolder`, `mode` (`"filename" | "content"`), `keywordMode: boolean` (phrase vs. keywords — see `search.rs`/`content_index.rs` above, default `false`), `results` (filename), `contentResults` (content, `ContentSearchResult[]`), `isSearching`, `error`. Deliberately a separate store rather than bolted onto `file-explorer.store.ts` — search has its own lifecycle, unrelated to navigation history. A module-level `latestRequestId` counter shared by `runSearch`/`runContentSearch` drops a slower, older response that resolves after a newer one (including across a mode switch mid-flight), so typing quickly — or toggling modes quickly — can't have a stale result clobber a fresher one. `scopeToFolder`, `mode`, and `keywordMode` are *not* reset by `clear()` — they're lasting preferences for the session, same as each other. `keywordMode` applies to **both** `runSearch` and `runContentSearch` (it's passed through as a plain extra argument to each, mirroring `mode`'s own dual applicability) — the user's request was explicit that this needs to work for "both file contents and normal search." **Closing the modal (`closeSearch()`) also does not clear the query/results** — only successfully opening a result does (via `useSearch.ts`'s `openResult`) — so reopening the modal resumes exactly where you left off, rather than forcing a retype every time; the mental model is "Escape dismisses, opening a result means you're done."

It's gated on `store.isOpen`: no backend calls fire while the modal is closed, even though the query/filters/results are still sitting in state (see above) — the effect depends on `isOpen` specifically so reopening the modal re-fires a fresh search rather than trusting stale results forever. Calling the triggering logic from two components would double-fire every debounced search (this bit it once already, historically, when it briefly lived inside `useSearch()` itself). `useSearchTrigger()` also re-runs the current query on `window` `focus`, gated the same way (`current.isOpen`). `withScope()` computes the effective `folder` filter sent to the backend: a manually-set `filters.folder` always wins; otherwise it's `currentPath` (read from `file-explorer.store.ts`) when `scopeToFolder` is on and the user isn't on "This PC".
- `lib/promoteExactMatch.ts` — pure function reordering an already-fetched result page so a name that exactly matches the query (or matches once its extension is stripped, so searching "billing" still promotes "billing.pdf" over a merely-containing "old_billing_report.pdf") moves to the front. Explicit user request: typing a full file/folder name and hitting Enter should reliably open *that* file, not whatever the backend's relevance/alphabetical ranking happened to put first. Client-side reorder over the existing capped result page, not a second backend query or a ranking change in `search.rs`/`content_index.rs`.
- `components/SearchModal.tsx` — the search overlay itself: backdrop + centered dialog (`animate-backdrop-in`/`animate-dialog-in`, the same pair `PromptModal`/`ConfirmModal` use, for visual consistency across every modal in the app), a two-row header (query input on its own row; mode/match/scope/filters controls on a second row below — see below), an optional inline `SearchFiltersFields` section, an error banner, a `role="listbox"` results list, and a footer hint bar (`↑↓ Navigate · Enter Open · Esc Close`).
- **The mode/match/scope/filters row uses text-labeled controls, not icon-only buttons with a `title` tooltip** (shaped via `/impeccable craft`). The original single-row layout packed the input plus four icon buttons (mode, phrase/keywords, scope, filters) into one line, each requiring a hover to discover what it did or which state was active — a real cost for these four specifically, since (unlike `Toolbar`'s back/forward/star, which are universally recognized glyphs) a magnifying-glass-with-a-document icon or a `Quote`/`ListChecks` pair has no self-evident meaning. `SegmentedToggle` (local to this file) renders `Names | Contents`, `Exact | Keyword`, and `Folder | Global` as two-sided pill switches with **both** states spelled out as text — clicking a side sets that state directly rather than flipping the current one, so a user can target the state they want without first checking which side is active. `Global` (not "Everywhere") deliberately reuses the user's own word for the scope toggle from the design conversation that shaped this. The `Filters` button gets the same treatment: visible word + its existing count badge + a `ChevronDown` that rotates 180° on open, instead of an unlabeled `SlidersHorizontal` icon. Per-segment icons (the old `Search`/`FileSearch`/`Quote`/`ListChecks`/`FolderIcon`/`Globe` glyphs) were dropped entirely rather than kept alongside the text — with real words present they were doing translation work the text does better, and removing them is part of what keeps three two-sided toggles plus a button fitting the modal's width. This moved the controls to their own row below the input (a genuine layout shift, not just a restyle) — packing four now-wider labeled controls onto the same line as the input would leave the input almost no room in a `max-w-xl` modal. The row uses `flex-wrap` as a safety net for narrow app windows rather than a hand-tuned fixed width. The footer's status line (previously `This folder · Names · Exact`) was removed as redundant now that the control row states the same thing persistently as text — the footer is keyboard hints only again. Implements the ARIA combobox/listbox pattern: the input is `role="combobox"` with `aria-activedescendant` pointing at the highlighted row's id, rows are `role="option"` — **result rows are never in the real tab order**; arrow keys move a "virtual" highlight while DOM focus stays on the input the whole time, matching how Spotlight/Raycast themselves behave and avoiding a heavier roving-tabindex implementation for something with a natural single-input-plus-list shape. A Tab focus trap (identical structure to `PromptModal`'s) still cycles between the input and the toggle/filter *controls* — it re-queries `input, button` fresh on every keydown, so it automatically expands to include the filter fields once that section opens rather than needing to know about them ahead of time. Highlight-reset (`useEffect` keyed on `search.results`/`search.contentResults`/`search.mode`, deliberately **not** on the derived `activeResults` array, which is a fresh reference every render and would re-trigger the effect — and therefore the `setHighlighted` inside it — on every single render) and scroll-into-view-on-highlight-change are the two other pieces of non-trivial state here.
- `components/SearchFiltersFields.tsx` — the Extension/Size/Date/Folder/Regex fields, extracted from the old `SearchBox.tsx`'s floating-popover `SearchFiltersPanel` with all the anchor/clamp positioning logic removed — `SearchModal` renders this as a plain inline disclosure in normal document flow, so there's no `position: fixed`/viewport-clamping math left to do at all. **Filters only apply to filename-mode search** — `SearchModal` disables the filters toggle in content mode (tooltip explains why) rather than half-wiring Extension/Size/Date/Regex support for Tantivy content search; those filters apply to the filename index's SQL `WHERE` clause and combining them sensibly with relevance ranking would need its own design pass, while the folder scope (the one filter content search *does* support, as a post-filter — see `content_index.rs`) carries over either way. `DateRangeFilter` stays colocated here (filter-specific, not extracted further) composing quick presets (Today/7d/30d/Year) with a visually-grouped native-date-input range control (`color-scheme: dark`).
- `components/Combobox.tsx` — a generic, styleable replacement for `<input list>` + native `<datalist>` (used by the Extension and Folder filter fields, sourced from a hardcoded common-extensions list and favorites + Quick Access paths respectively; also still supported, via `autoFocus`, as the `suggestions` renderer `PromptModal` falls back to, even though nothing currently sets that field — see `PromptModal`'s own entry above). Native datalist options render as unstyleable browser chrome — fine functionally, but can't be visually designed to match the rest of the app. Closes on the input's own `onBlur`, **not** a `window` click listener — a click-outside listener would need the popover it lived inside (the old `SearchFiltersPanel`) to *not* already stop click propagation at its own root, which it did; the standard fix instead is `onMouseDown={(e) => e.preventDefault()}` on each option (keeps the input focused so blur doesn't fire before the click's `choose()` runs). Its own Escape handler calls `e.stopPropagation()` before closing the suggestion list — needed since it's reused inside `PromptModal` (and now `SearchModal`'s filters section), both of which have their own Escape-to-cancel/close on the surrounding form; without stopping propagation, closing just the dropdown would also incorrectly dismiss the whole modal. **The dropdown's displayed label can differ from the actual option value** via an optional `getOptionLabel` prop (defaults to identity) — the Folder filter field uses this to show `"Documents (C:\Users\carlo\Documents)"` instead of the bare full path (`SearchFiltersFields.tsx`'s `folderOptionLabel`, built from `lib/path.ts`'s `basename`), since a dropdown full of long, similarly-prefixed paths is hard to scan. The input's own committed `value` is always the real path either way — only the suggestion list's text changes — and filtering matches against both the raw option and its rendered label, so typing "Documents" still surfaces a suggestion stored as a full path.
- `components/IndexStatusBadge.tsx` — the "Indexing… N" pill, a floating `fixed`-position badge (not inline in the `Toolbar`, unlike Phase 2's original placement).
- `lib/entryType.ts` — `entryTypeLabel(entry)`, the "Type" column/grouping value: the extension itself uppercased ("PNG", "TXT") or `"Folder"`/`"File"` — deliberately not an OS-style friendly-name dictionary ("PNG Image", "Text Document"), same honesty-over-polish tradeoff as `PreviewPane`'s Office-preview banner. Its own tiny `extensionOf()` (dot-at-index-0 is a dotfile, not an extension) duplicates `previewKind.ts`'s identical four-line helper rather than importing it — the two serve genuinely different purposes (broad preview-renderer buckets vs. a literal per-extension label) and four lines isn't worth a cross-file dependency for.
- `lib/sortEntries.ts` — `sortEntries(entries, key, direction)`, `key: "name" | "type" | "size" | "modified"`. **Folders always sort before files regardless of key or direction** — matches Explorer's own default (reversing direction only reverses order *within* each group, it never interleaves folders and files) and this project's "Native Feel" core principle. "Name" comparison uses `localeCompare` with `numeric: true` so `file2.txt` sorts before `file10.txt`.
- `lib/groupEntries.ts` — `groupKeyFor(entry, groupBy, nowMs?)` (`groupBy: "none" | "type" | "modified" | "size"`) and `toDisplayItems(entries, groupBy)`, producing a flat, already-interleaved `({ kind: "header"; label } | { kind: "entry"; entry })[]` — both `EntryTable` and `EntryGrid` render group headers by walking this list and noticing when the group key changes from the previous item, no nested group tree needed. Assumes the input is already ordered so same-group entries are contiguous (guaranteed by the store's `organizeEntries`, see above). Date buckets (Today/Yesterday/This Week/This Month/Earlier) and size buckets (Empty/Small `<1MB`/Medium `<100MB`/Large `≥100MB`) are simplified from Explorer's own finer tiering — stated v1 ceilings. `nowMs` defaults to `Date.now()` but is an explicit parameter specifically so tests can pass a fixed reference point instead of mocking global `Date`. Size grouping buckets every folder under its own `"Folder"` key rather than lumping them into `"Empty"` — folder sizes aren't meaningfully tracked (`lib/format.ts`'s `formatSize` already shows `""` for a directory), so an `"Empty"` bucket mixing real empty files with every folder would be misleading.
- `components/EntryGrid.tsx` — the Medium/Large icon-grid view, a `size: "medium" | "large"` prop away from being two near-duplicate components. Same prop shape as `EntryTable` (selection/`onOpen`/`onSelect`/`onContextMenu`/`onDragPaths`/`onDrop`/`cutPaths`) so `FileExplorerView.tsx` swaps between them with no selection/DnD/context-menu wiring changes. Renders one CSS grid for the whole listing (not one grid per group) — a group-header item uses Tailwind's `col-span-full` to break out of the tile columns for that row, then tiles resume flowing normally after it, avoiding the need to re-nest `toDisplayItems`'s flat list into a group tree just for rendering. Both the group-header `<p>` and the grid container `<div>` itself carry their own `onClick`/`onContextMenu` (not just the outer scrollable container's) — a right-click landing in the *gap* between tiles, or directly on a group-header label, lands on one of these elements rather than the outer container, and both are conceptually "empty space" that should open the background menu, not silently do nothing. A trailing `min-h-24 flex-1` spacer after the grid guarantees there's always some background left to right-click even when tiles exactly fill (or overflow) the view — same fix applied to `EntryTable`.
- **Image files render their actual content as the tile's icon (`EntryTile`'s `showImage` branch, `previewKind(entry.name) === "image"`) — everything else keeps the generic Folder/File glyph.** Reuses existing infrastructure with zero new backend work: `fileExplorerService.assetUrl()` (the same Tauri asset-protocol call `PreviewPane` already uses) and the already-built `previewKind()` categorization. `object-contain` (not `cover`) shows the whole image scaled to fit, matching Explorer/Finder's own icon-thumbnail convention rather than cropping. `loading="lazy"` + an `onError`-driven `imageFailed` state (falling back to the generic icon, not a broken-image glyph, for a corrupt file or a decode failure) round it out. **Explicit, stated ceiling**: this loads each image at full resolution through the asset protocol — no resize/cache step, which is the separate, already-unscheduled `# Thumbnail System` (`image` crate, FFmpeg, a real thumbnail cache) Phase 4's plan explicitly declined to build. `loading="lazy"` keeps this workable for typical folders (only near-viewport tiles decode); a real cache is the natural upgrade path if a large real-world photo folder proves this too slow. Video/PDF tiles deliberately still show the generic icon — a representative frame/page needs real extraction work, not a zero-effort reuse of existing infra, so they're out of scope for this pass.
- `components/ViewMenu.tsx` — the `Toolbar`'s "View, sort, and group" dropdown. Reuses `ContextMenu.tsx`'s own measure-then-clamp positioning technique (`useLayoutEffect` + `getBoundingClientRect`, corrected before paint) — but is **not** a shared abstraction with `ContextMenu`; considered and declined, since the two have different content shapes (per-entry actions vs. view-settings radio groups) and forcing a shared popover component for two dissimilar call sites right now would be the premature generalization this codebase already avoids elsewhere. Every list item shows a checkmark on the active choice (`Check` icon, transparent when inactive rather than a hidden element, so the layout doesn't shift) — the same "state visible without hovering" standard applied to every toggle since the `SearchModal` redesign. `Toolbar.tsx` owns the anchor-position state and its own window-click/`resize` close listener, the same self-contained pattern `SearchModal` already uses for its result-row context menu, rather than pushing this transient UI state into the store.
- `components/PreviewPane.tsx` — the resizable right-side panel from Phase 4, mirroring `Sidebar`'s resizable left-side one (same handle-as-sibling structure, same `useDragResize` mechanics via `usePreviewResize`). Toggled by a new `Toolbar` button (`onTogglePreview`/`previewOpen`, persisted in `file-explorer.store.ts` the same lasting-preference way as `sidebarWidth`) — **not** a keybinding; `plan.md`'s `Space` → "Quick Preview" shortcut waits for Phase 6's proper hotkey system, per this project's own stored convention against ad hoc `onKeyDown` checks. `FileExplorerView.tsx` derives the entry to preview inline (`selectedEntries.length === 1 ? selectedEntries[0] : null`, deliberately **not** also excluding a selected folder — `PreviewPane` has its own explicit "Folders don't have a preview" empty state, which is more informative than falling back to the generic "select a file" message a folder-exclusion would produce) — no new plumbing needed in `useFileExplorer.ts`, since `selectedEntries` already existed there. Dispatches on `previewKind(entry.name)`: Image/Video/PDF render straight from `fileExplorerService.assetUrl()` with no backend round-trip at all; Markdown/Text/Office fetch via `previewText` (a `Loader2` spinner while in flight, matching `SearchModal`'s existing loading-state pattern) and render either through `react-markdown` or a plain `<pre>`; Archive fetches via `listArchiveEntries` and renders a flat name/size/is_dir list. The fetch effect is keyed on `[entry, kind]` and gated on `!entry.is_dir` up front — no backend calls fire for a directory selection, an empty selection, or an already-renders-natively image/video/pdf.

Icons are `lucide-react` throughout (finally realizing DESIGN.md's long-standing "use Lucide" spec — emoji were a placeholder), **except file rows/tiles, which use the real Material Icon Theme SVG set via `lib/fileTypeIcon.tsx`** (see above) — lucide's own file glyph only survives there as an unreachable last-resort fallback. Each interactive component repeats its own `focusRing`/`itemClass` Tailwind string rather than importing a shared one, since the `ring-offset-*` color has to match that component's own background — this is normal Tailwind practice, not an oversight; don't "fix" it into a single shared constant without also parameterizing the offset color.

## Non-obvious gotchas (cost real debugging time — don't rediscover these)

- **`useFileExplorer()`'s startup `init()` (loads quick-access dirs/drives, navigates to This PC) must be idempotent, not just "conventionally called once."** It was written assuming exactly one caller (`FileExplorerView`), until `SearchModal` also started calling `useFileExplorer()` (to reach `openFileLocation`/`renameEntry`/etc. for its context menu) — both mount at app startup, so `init()` now fires from two places. `file-explorer.store.ts` gained an `initialized` flag set synchronously (before the first `await`) at the top of `init()`, so a second call arriving before the first one's async work even resolves still sees it and bails, instead of double-fetching `quickAccessDirs`/`listDrives` and double-navigating to `THIS_PC`. Any future hook that also needs `useFileExplorer()`'s action functions hits this for free — the guard lives in the store, not in the call site.
- **`tauri.conf.json` → `app.windows[].dragDropEnabled` must be `false`.** Tauri's window has its own native OS drag-drop hook (for detecting files dragged in *from* the OS), on by default, and it intercepts native drag events at the WebView2/OS layer *before* they reach the page's own HTML5 `dragstart`/`dragover`/`drop` — including drags that originate entirely inside the webview. Without this set to `false`, in-app row dragging silently doesn't work.
- **Tauri plugin permissions often need an explicit scope, separate from the permission bit.** `opener:allow-open-path` alone does *not* allow opening files — `open_path`'s Rust implementation checks a scope (allowed paths) independently of the permission, and an empty scope always denies. Capability entries needing a scope use the object form, not a bare string: `{"identifier": "opener:allow-open-path", "allow": [{"path": "**"}]}`. See `src-tauri/capabilities/default.json`.
- **`useDropTarget`'s `onDragOver` must set `e.dataTransfer.dropEffect` explicitly.** `preventDefault()` alone permits the drop logically, but the cursor still shows "no-drop" unless `dropEffect` is set to match (`"move"` or `"copy"`).
- **Dropdowns/menus positioned `absolute` inside an `overflow-hidden` ancestor get visually clipped**, even though they're in the DOM. `ContextMenu` and the breadcrumb ellipsis dropdown both use `position: fixed` anchored via `getBoundingClientRect()` instead. **That alone isn't enough** — anchoring to the raw click/button coordinates with no regard for viewport edges clips the menu when it opens near the bottom or right edge (confirmed live: `ContextMenu` did exactly this). The fix (`ContextMenu.tsx`) is measure-then-correct: render at the raw anchor position, then in `useLayoutEffect` (not `useEffect` — this must happen before paint or the correction visibly jumps) measure the element's actual rendered size via a ref and clamp `top`/`left` against `window.innerWidth`/`innerHeight`. `SearchModal`'s filters section sidesteps this class of bug entirely rather than needing the same fix — it's a plain inline disclosure in normal document flow inside an already-centered dialog, not a separately-positioned floating popover, so there's no anchor point to clamp in the first place. The breadcrumb ellipsis dropdown doesn't do the clamp fix yet (lower risk in practice since it's anchored inside the address bar, not at a window edge, but the same class of bug could still hit it with a long enough hidden-segment list).
- **A Tauri command's own parameter names (as opposed to a struct's serde fields) are matched camelCase by default, even when the Rust side is snake_case.** `search_files`/`search_content`'s new `keyword_mode: bool` parameter, invoked from JS as `{ query, filters, keyword_mode }`, failed at runtime with `invalid args \`keywordMode\` for command \`search_files\`: command search_files missing required key keywordMode` — Tauri's IPC argument matcher camelCases each `#[tauri::command]` parameter name unless the command opts out via `#[tauri::command(rename_all = "snake_case")]`, which none of this app's commands do. This is a *different* mechanism from `SearchFilters`'s own struct fields (`min_size`, `modified_after_ms`, etc.), which stay snake_case verbatim on both sides via plain serde with no `rename_all` — those matched fine because struct deserialization isn't run through the same camelCasing. Fixed by sending `keywordMode` (not `keyword_mode`) as the invoke() object key in `file-explorer.service.ts`, while keeping the Rust-side parameter itself snake_case.
- **Draggable table rows need `select-none` on the whole `<tr>`, not just one cell.** Starting a drag gesture from a cell without it lets the browser begin a native text-selection drag instead of firing the row's own `dragstart`.
- **`copy_entry`/`move_entry` run every destination through `unique_destination()` before touching disk.** Without it, copying/moving into a folder that already has a same-named entry either silently overwrites it or (when source == destination, e.g. copying into the file's own folder) fails with a Windows sharing-violation error. It auto-appends `" (1)"`, `" (2)"`, etc., matching Explorer/Finder — this is unrelated to the Phase 5 "Duplicate Detection" feature (that's content-hash based, across the whole drive; this is filename-collision handling for a single operation).
- **The app root (`FileExplorerView.tsx`) sets `onContextMenu={(e) => e.preventDefault()}`** to suppress the native browser context menu everywhere by default. `EntryTable` rows override this locally with their own `preventDefault()` + custom `ContextMenu`. Any new area that wants a custom right-click menu needs to do the same; anywhere else, right-click silently does nothing (intentional, not a bug).
- **Prefer a documented Win32 API over an informal `rundll32`/PowerShell+COM shell-out for anything that needs to reliably open a real OS dialog.** Both were tried for `open_with_dialog`/`show_properties` and both silently no-op'd (clean exit code, zero output, no visible window) — no error to catch, indistinguishable from the feature doing nothing. `ShellExecuteExW` (see above) fixed it outright. If a future feature still needs to spawn a helper process that shows GUI, don't use `CREATE_NO_WINDOW` on it (it can suppress the resulting window's focus/foreground activation) — use the tool's own hidden-window flag if it has one (e.g. PowerShell's `-WindowStyle Hidden`, which only hides its own console).
- **A `LIKE` query against an FTS5 external-content table's column does not get accelerated by the trigram tokenizer** on the SQLite version this project's `rusqlite` bundles — it was tried first (`files_fts.name LIKE '%x%'`) and measured *slower* than just scanning `files.name` directly, while also (separately, and worse) returning wrong result counts for real substrings that definitely existed in the index. The correct, version-independent way to substring-search a trigram-tokenized FTS5 table is a **quoted phrase `MATCH`** (`files_fts MATCH '"chrome"'`) — this is what actually uses the trigram index and returns correct results. Confirmed by cross-checking `MATCH` result counts against a known-correct `files.name LIKE` baseline on the live index before trusting it.
- **A SQLite `REGEXP` (or any per-row scalar function) that compiles anything expensive must cache the compiled result across invocations**, not recompile it inside the function body. `create_scalar_function` is called once *per row*, and a query re-invokes it with the *same* pattern for every row in a single query — compiling a `regex::Regex` fresh each time turned one query into a 300+ second scan against the real ~1.5M-row index. A `thread_local!` cache of just the most-recently-compiled pattern fixed it (a query never interleaves two different patterns on the same thread), cutting the same query to ~1.2s.
- **An expensive one-time SQLite migration (e.g. `INSERT INTO fts_table(fts_table) VALUES('rebuild')` to backfill an FTS5 index for pre-existing rows) must not run inside a function called from Tauri's `setup()` hook.** `setup()` runs synchronously before the window appears — a 118-second rebuild measured against the real ~1.5M-row index would make every existing user's *next app launch* hang with no visible window and no explanation. `database::open()` only does cheap, size-independent schema/trigger creation; `database::backfill_fts_if_needed()` (the row-count-comparison + potentially slow rebuild) is a separate function, called only from the indexer's background thread. A brand new install never pays this cost at all — `files` is empty when the schema is first created, so the triggers alone keep `files_fts` current as rows stream in.
- **A number typed into an HTML `<input type="number">` is not actually bounded by anything unless you enforce it yourself.** `min`/`step` attributes are presentational hints, not runtime constraints on the JS value you read from `onChange`. `SearchFilters`' `min_size`/`max_size` are `u64` on the Rust side — a negative value typed into the size filter serializes fine as JSON but fails to *deserialize* into `u64` at the Tauri IPC boundary, rejecting the whole `invoke()` call and surfacing as a search error. `SearchFiltersFields.tsx`'s `parseNonNegative()` clamps and floors before it ever reaches `set()`. Found by testing the actual deserialization path (`serde_json::from_value` against a value shaped like what the frontend sends) rather than assuming an `Option<u64>` "just works" — the Rust-side unit tests never construct `SearchFilters` through JSON, so they can't catch this class of bug.
- **Every input that can trigger a search must be debounced consistently, not just the main query box.** `useSearch.ts` originally only debounced `search.query`, while `search.filters` (the extension/folder/regex text filters) went straight into the effect's dependency array — meaning every keystroke in a filter field fired an immediate, undebounced backend call, unlike typing in the search box itself. Both now go through `useDebouncedValue`.
- **A `notify` watcher must be registered *before* the work that could take a while, not after.** Registering it (and starting to drain its events) only once the entire multi-drive initial scan finishes meant a file created mid-scan fell into a real gap and was never indexed until the next full app restart happened to re-cover it — confirmed live by creating a file seconds after launch and finding it missing from search. The fix has two parts, both required: `start_watchers()` runs before scanning (queues events harmlessly if nothing's draining them yet — notify's OS-level watch and the channel itself don't need a reader to exist first), and `drain_events()` runs on its own thread *concurrently* with the scan (via a shared `Arc<Mutex<Connection>>`), not sequentially after it — otherwise a live create still wouldn't show up until the whole scan (minutes, for a real drive) completed, which reads as "still broken" even though nothing is technically lost anymore.
- **An exclusion check on a multi-path filesystem event (like a rename) must be applied per-path, not to the event as a whole.** `apply_event` used to skip the *entire* event if any of its paths were excluded — which broke deletion, since deleting a file moves it into `$RECYCLE.BIN` (excluded) and `notify` reports that as a rename (old real path → new excluded path). Skipping the whole event meant the old, non-excluded path never got deleted from the index, so a deleted file kept showing up in search forever. Verified with a real Recycle Bin delete (`Shell.Application`'s `InvokeVerb("delete")` via PowerShell — the actual Explorer mechanism, not a plain `rm`, which just permanently removes the file and would've masked this bug entirely).
- **Vite silently base64-inlines any imported asset under its default 4kb threshold — including inside `import.meta.glob`.** `lib/fileTypeIcon.tsx`'s glob over all 1250 `material-icon-theme` SVGs looked harmless (most are just a few hundred bytes to a couple KB), but ~1225 of them landed under that threshold, so Vite embedded their actual image bytes as base64 text straight into the main JS bundle instead of leaving them as separate files — measured: a real production `vite build` came out to 1.7MB (366KB gzip) for the main chunk. The fix is the glob's `query: "?url&no-inline"` (not just `"?url"`, which still respects the inline threshold) — forces every matched SVG to stay a real, separate file that an `<img>` only loads when actually rendered on screen, instead of being parsed/evaluated as part of every app startup regardless of whether that file type ever appears. Confirmed via a before/after production build: main chunk dropped to 706KB (180KB gzip) after adding `&no-inline`, with the 1250 SVGs correctly showing up as separate `dist/assets/*.svg` files instead.
