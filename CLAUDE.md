**`create_dir`/`create_file`/`rename_entry` (`fs_ops.rs`) all reject a name collision up front instead of silently corrupting or erroring cryptically.** Three real, reported bugs, same root cause: Rust's `fs::write`/`fs::rename` have no "fail if the destination exists" mode. `create_file` used `fs::write(path, [])`, which silently truncated an existing file with that name to empty (no error, no visible new entry, which read to the user as "nothing happens" but was actually quiet data loss); naming it the same as an existing *folder* surfaced Windows' raw `"Access is denied. (os error 5)"` instead, since `CreateFileW` denies write access to a directory rather than reporting a collision. `rename_entry` was worse â€” Rust's Windows implementation of `fs::rename` always calls `MoveFileExW` with `MOVEFILE_REPLACE_EXISTING`, so renaming a file onto an existing name silently **deleted that other file outright**, no error, no trace, confirmed live by a real rename that made an existing file "just disappear." All three commands now `Path::new(&path).exists()`-check before touching disk and return one friendly `"A file or folder with that name already exists."` regardless of what's already there â€” covered by `fs_ops.rs` tests asserting the existing file's content survives untouched (for both create-collision cases and the rename-collision case) and that file-vs-file, file-vs-folder, and rename-vs-existing collisions all produce the same clean error (surfaced inline by `PromptModal`, same as any other validation failure). `copy_entry`/`move_entry` don't need this guard â€” they already run every destination through `unique_destination()` to auto-number instead of colliding (see below); `rename_entry` deliberately does *not* get that same auto-numbering treatment, since silently renaming to a different name than what the user typed, without telling them, would be its own kind of confusing.

**Cutting a file and pasting it back into the same folder it's already in is a no-op, not a numbered duplicate.** `pasteIntoCurrent()` used to call `moveEntry` unconditionally, which â€” same as any other destination collision â€” went through the backend's `unique_destination()` and produced a `" (1)"`-suffixed copy instead of actually moving anything, which is correct for a *copy* pasted into its own folder (Explorer does the same: Ctrl+C, Ctrl+V in place makes a duplicate) but wrong for a *cut*, where there's nowhere to move to. `pasteIntoCurrent()` now filters `clip.paths` to drop any path whose `dirname()` already equals `store.currentPath`, but only when `clip.op === "cut"` â€” a copy pasted in place still goes through and still numbers, unchanged. If filtering leaves nothing to move, the function returns immediately without calling the backend, refreshing, or clearing the clipboard, so the cut is still live and pasteable somewhere else afterward.

## Commands

- `npm run tauri dev` â€” run the full desktop app (Rust + webview). **Capability/`tauri.conf.json` changes require a full stop + restart of this process**, not just a browser refresh â€” Tauri compiles capabilities and window config into the Rust binary at build time, so Vite's HMR never picks them up.
- `npm run dev` â€” Vite dev server alone (frontend only; `invoke()` calls will reject without a real Tauri context).
- `npm run build` â€” `tsc` then `vite build`.
- `npm test` / `npx vitest run` â€” frontend unit tests for the pure `lib/` helpers (`path`, `previewKind`, `tabs`, etc. â€” the same "test the non-trivial pure logic, not the components" precedent throughout).
- `cargo test` (from `src-tauri/`) â€” Rust unit tests in `fs_ops.rs` (quick access dirs, drive enumeration, recursive copy, move), `database.rs` (upsert/update/delete against a temp SQLite file), `indexer.rs` (scan against a temp directory tree, notify-event application, excluded-directory pruning for both the scan and the live watcher), and `search.rs` (substring/escaping, each filter individually and combined, result cap â€” all against a temp SQLite file seeded through the same `database::open()`/`upsert_batch()` path production code uses, so the FTS5 triggers are exercised for real, not mocked).

## Architecture

Tauri app: React/TypeScript frontend calls Rust commands via `invoke()`; the frontend never touches the filesystem directly. All filesystem work lives in `src-tauri/src/fs_ops.rs` (single module â€” `home_dir`, `quick_access_dirs`, `list_drives`, `list_dir`, `create_dir`, `create_file`, `rename_entry`, `delete_entry` (via the `trash` crate, not `fs::remove_*`), `copy_entry`/`copy_dir_all`, `move_entry` (renames, falls back to copy+delete across volumes), `open_with_dialog`, `show_properties`). `dirs` crate resolves platform folders; drives are enumerated by probing `A:`â€“`Z:` (Windows-specific, not yet cross-platform).

`open_with_dialog`/`show_properties` invoke the real native Windows dialogs via `ShellExecuteExW` (`windows` crate, Windows-only dependency in Cargo.toml) with the `"openas"`/`"properties"` verbs and `SEE_MASK_INVOKEIDLIST` (required for `"properties"` to work), instead of reimplementing an app-picker or a General/Security/Sharing panel â€” see plan.md's File Operations section. This is the actual public Win32 API Explorer itself uses. Two informal alternatives were tried first and both silently no-op'd (exit 0, no output, no visible window) depending on the calling process's context: `rundll32 shell32.dll,OpenAs_RunDLL` (an undocumented internal DLL export, not a supported API) and PowerShell + `Shell.Application` COM `InvokeVerb('properties')` (COM apartment state sensitive â€” `InvokeVerb` needs STA, PowerShell defaults to MTA). Don't go back to either without a concrete reason.

**Indexing (`database.rs` + `indexer.rs`)**: on app startup (`lib.rs`'s `setup()` hook), one `std::thread` is spawned that owns a SQLite connection (`rusqlite`, bundled), shared as `Arc<Mutex<Connection>>` between the scan and the live-event drain thread (see below) â€” nothing else writes to the DB, so this Mutex is the only writer-side synchronization needed. `indexer::spawn()` scans the user's home directory first (via `fs_ops::home_dir()`), then every drive â€” so useful results land quickly instead of waiting for a full drive to finish; the home dir gets walked twice total (once standalone, once again inside its drive's full walk) since nothing tracks already-covered paths, an accepted tradeoff since upserts are idempotent. Each `scan_drive()` call lists the root's immediate children on the calling thread, then walks each child subdirectory's subtree **in parallel via `rayon`** (`par_iter().for_each`), with the shared `Mutex<Connection>` so parallel workers serialize on the (fast, batched) writes. `EXCLUDED_DIR_NAMES` (`node_modules`, `.git`, `.cache`, `$RECYCLE.BIN`, `System Volume Information`, plus package-manager/dependency caches â€” `.cargo`, `.rustup`, `.npm`, `.nuget`, `.gradle`, `.m2`, `.venv`, `venv`, `__pycache__`, `site-packages` â€” build output â€” `target`, `dist`, `build`, `.next` â€” and, since it dominated the Phase 5 Recent Files feature's results with junk, `AppData`) is checked via `walkdir`'s `filter_entry` so those trees are pruned entirely â€” never even listed â€” rather than walked and discarded; it's a hardcoded list, not a user setting (add a Settings toggle in Phase 6 if that's ever needed). **`AppData` excludes Roaming, Local, and LocalLow all at once** â€” all three live directly under a folder literally named "AppData" in every user profile, so this single name-based entry (case-insensitive, like every entry in this list) prunes the whole tree: browser profiles (Chrome's `Cookies`/`Cookies-journal`/network-state journal files), npm/yarn/pip caches under the user profile, `%TEMP%`, and every other app's private scratch data. Confirmed live: exactly this junk (Cookies files, npm debug logs, stray `%TEMP%` files) was what a real scan surfaced at the top of the Recent Files list once that feature shipped â€” none of it is ever something a user searches for by name. This also made the old dedicated `com.carlo.schlag` exclusion entry redundant (this app's own data directory lives under `%APPDATA%\com.carlo.schlag` too) â€” removed rather than left as a second, overlapping entry. **`EXCLUDED_DIR_NAMES` also covers a handful of individual, singleton *files* now** (not just directories) â€” `hiberfil.sys`/`pagefile.sys`/`swapfile.sys` (Windows' own virtual-memory backing files) and `DumpStack.log.tmp`, all living at a drive's root and among the most frequently-rewritten files on the entire system; confirmed live sitting at the very top of a real Recent Files list, ahead of anything the user actually did. `is_excluded()` already runs against every entry (file or directory) the scan's root-level loop and `walk_subtree`'s `filter_entry` see, so these fit the same list rather than needing a parallel one. **`.bun`/`.dotnet`/`.docker`/`.android`/`.expo`/`.ollama` joined the same package-manager/SDK-cache group as `.cargo`/`.npm`/`.nuget`** â€” confirmed live via a real filename search for "cookies" that surfaced `.bun`/`.dotnet` package-cache hits alongside legitimate project files (a real `Cookies` folder, vendor PHP/JS code) before this; same evidence bar as every other addition here, not speculative. **`.ssh` is excluded for a different reason than the rest of this list â€” security, not noise.** A general-purpose file explorer surfacing SSH private key filenames, `known_hosts`, or `authorized_keys` in Recent Files or search results is a real exposure regardless of how often those files actually change; this entry exists independent of whether `.ssh` would otherwise qualify as "noisy."

**This exclusion mechanism lives entirely in the indexer/scanner (`indexer.rs`), not as a filter bolted onto any one feature** â€” `scan_drive`/`walk_subtree` decide what ever gets *read and written* into the SQLite `files` table in the first place, and `apply_event` applies the identical check to live filesystem events, so an excluded path never exists as a row at all, for any consumer. `search_files` (filename search), `search_content` (via `queue_content_event`, only ever called for entries that already passed this same walk's exclusion check), and `recent_files` all read from that same table â€” none of them re-implement or duplicate the exclusion logic themselves. Verified directly against a real ~1.27M-row index: after this shipped, a filename search for `"cookies"` returned only genuine files (a real `C:\Users\carlo\Cookies` folder, vendor package code) and zero hits under `AppData`; a search for `"hiberfil"` or `"evtx"` returned zero results entirely, confirming exclusion holds for search, not just the Recent Files UI that first surfaced the problem.

**`content_index_state` (the Tantivy content-index's own bookkeeping table, see below) can drift out of sync with `files` independently, and needs its own separate reconciliation pass â€” `prune_stale_entries` re-deriving removals from `files`'s stale list isn't enough.** Found live, not theoretical: after the `AppData` exclusion shipped, `prune_stale_entries` correctly queued `ContentEvent::Remove` (over the content-indexer thread's in-memory channel) for ~500k newly-excluded paths â€” but this app restarted (Tauri's dev-mode file watcher rebuilding on a later source change, an ordinary occurrence in this session) before that backlog fully drained. Those in-memory removal requests are gone once the process exits; since `files` already had no row for those same paths by then (the batched SQLite delete had already committed), `prune_stale_entries`'s *next* run never re-derives that removal either â€” it only ever looks at what's *currently* in `files`. Left over 50,000 `content_index_state` rows permanently orphaned under `AppData`, confirmed by directly querying the table while the content-indexer thread sat completely idle (CPU time unchanged over a 5-second window) â€” not a slow drain, a real gap with no self-healing path under the old design. `indexer.rs`'s `prune_stale_content_state()` fixes this by reconciling `content_index_state`'s **own** paths directly against `path_is_excluded()`/existence, the same way `prune_stale_entries` does for `files`, rather than depending on that one-shot side channel ever having succeeded â€” called right after `prune_stale_entries` in `spawn()`. Its removals still go through the same `ContentEvent::Remove` channel (so `content_index::remove_path()` still runs against the live Tantivy writer), but the `content_index_state` SQLite side is now independently self-healing. **`run_content_indexer`'s `ContentEvent::Remove` handling was also batched** (`pending_removals: Vec<String>`, flushed via `database::delete_content_indexed_mtime_batch()` at the same points `writer.commit()` already fires, plus once more after the channel closes) â€” the old per-event `delete_content_indexed_mtime` call was the exact same "individual delete = an order of magnitude slower at scale" pattern already fixed once for `prune_stale_entries`'s `files`-table deletes, and `prune_stale_content_state` can legitimately queue tens of thousands of removals in one shot after any future exclusion-rule change. The old single-path `delete_content_indexed_mtime` function was removed outright once nothing else called it, rather than left as dead code.

**A second, separate exclusion list, `EXCLUDED_ROOT_DIR_NAMES` + `is_excluded_root_dir()`, covers whole OS/application-install trees â€” `Windows`, `Program Files`, `Program Files (x86)`, `ProgramData`, `Recovery`, `PerfLogs`, `$GetCurrent` (Windows Update's staging folder), and `Documents and Settings` (the legacy Vista-era compatibility symlink to `Users`) â€” but matched *only* when the name is a drive's own top-level folder, unlike `EXCLUDED_DIR_NAMES`'s anywhere-in-the-path matching.** This distinction is deliberate, not an oversight: `node_modules`/`AppData` are names no real user would ever give their *own* folder, so matching them at any depth is safe, but `Windows`/`Program Files` are real (if uncommon) names a user could legitimately nest inside their own project (a cross-platform build's own `windows/` folder, say) â€” matching those anywhere would risk silently excluding real user content the same careless way the `node_modules`-substring false-positive above already warns against. `is_excluded_root_dir()` instead checks specifically whether a path's first component after the drive prefix (`Windows` in `C:\Windows\System32\...`) is one of these names, so a deeply-nested user folder that happens to share a name is untouched, while the drive's own real system folder â€” at any depth beneath it â€” is fully excluded. Confirmed live against a real machine's actual `C:\` root (`$GetCurrent`, `Program Files`, `Program Files (x86)`, `ProgramData`, `Recovery`, `Windows` all present) once `AppData`'s own noise was cut: `C:\Windows` alone (System32 DLLs, `winevt` event logs, `catroot2`, scheduled Tasks) immediately took over the top of Recent Files instead â€” the same class of problem, one level up the drive tree. Wired into the same three places `EXCLUDED_DIR_NAMES` is: `scan_drive`'s root-level loop (alongside `is_excluded()`), and folded into `path_is_excluded()` (used by both `apply_event` and `prune_stale_entries`) so live events and the retroactive prune of an already-populated index cover it too â€” `walk_subtree` doesn't need its own separate check, since it's only ever invoked on a subdirectory that already passed `scan_drive`'s root-level filter, so an excluded root dir's subtree is never walked into in the first place. **Adding an exclusion doesn't retroactively clean an already-populated `index.db`** â€” the scan only ever adds/updates rows, so `prune_stale_entries()` (below) was extended to also delete any already-indexed row that's still genuinely present on disk but now matches `path_is_excluded()`, not just rows for files that no longer exist; otherwise a user upgrading past this change would keep seeing the exact junk it was meant to fix, forever, since nothing else ever revisits a row once it's in the DB. The `notify` watcher doesn't get this pruning for free â€” it recursively watches whole drives regardless of what the initial scan skipped â€” so `apply_event` separately checks every path component of an incoming event against the same list (`path_is_excluded`) before touching the DB; without it, live filesystem activity inside an excluded folder (e.g. a dev server writing into `node_modules/.vite`) would silently re-populate it. Confirmed live: a real scan initially showed 2,000+ false-positive `node_modules` hits that turned out to be filenames merely *containing* that substring (bundler chunk names, `.asar` bundles) â€” the exclusion itself only ever matches an exact path *component*, not a substring. **This check is per-path, not per-event** â€” `apply_event` used to skip a whole multi-path event if *any* of its paths were excluded, which broke deletion: deleting a file (via Explorer or `fs_ops::delete_entry`'s `trash` crate) moves it into `$RECYCLE.BIN` rather than removing it outright, which `notify` reports as a rename (old real path â†’ new `$RECYCLE.BIN` path) â€” and since the new path is excluded, the *whole event* got skipped, so the old path never got deleted from the index and the file kept showing up in search forever. Verified live: sent a real file to the Recycle Bin via `Shell.Application`'s `InvokeVerb("delete")` COM call (the actual mechanism Explorer uses, not a plain `rm`) and confirmed it dropped out of search within seconds after the fix.

`start_watchers()` registers a recursive `notify` watcher per drive **before** the initial scan starts (not after), and `drain_events()` (applying create/modify/remove/rename events) runs on its **own thread concurrently with the scan**, not sequentially after it. Both of these were fixed after a real, reachable bug: registering watchers only after the full scan meant a file created *during* a long scan fell into a permanent gap (the watcher didn't exist yet when the OS-level create event fired) and never got indexed until the next full app restart happened to re-cover it. Draining sequentially-after-scan (an earlier, partial fix) turned that into "doesn't show up until the *entire* scan finishes," which for a real multi-million-file `C:\` can be minutes â€” still indistinguishable from broken to a user who just added a file. Verified live: created a file mid-scan (confirmed via log â€” "initial scan complete" hadn't printed yet) and it was searchable within ~3 seconds via the concurrent drain thread. Deliberately **not** `tokio`-based: rusqlite/walkdir/notify are all synchronous, and nothing else in this codebase is async â€” bare OS threads (plus `rayon` for the one CPU/IO-bound parallel step) are simpler and consistent with `fs_ops.rs`'s style. Progress is exposed via `IndexStatus`, in-memory atomics (not a live `SELECT COUNT(*)`) polled by the `index_status` command and the frontend's `useIndexStatus()` hook. DB lives at `{app_data_dir}/index.db`. Walkdir/notify errors (permission-denied, e.g. leftover access-denied subfolders) are logged via `tracing::warn!` and skipped, never propagated.

**Watching a whole drive via `ReadDirectoryChangesW` (what `notify` uses on Windows) can silently drop events under load** â€” confirmed live: deleting a file via the `trash` crate produced *no* notify event at all despite the file genuinely being gone, while the same delete via `Shell.Application`'s `InvokeVerb("delete")` COM call did produce one. This isn't a bug in this codebase's code, it's an inherent limitation of recursively watching an entire drive with a lot of background churn â€” and this app's own `index.db`/`index.db-wal` writes were adding to that churn unnecessarily (every DB write is itself a filesystem change the recursive watcher picks up and re-indexes as a normal file, a self-referential loop) â€” covered today by the broader `AppData` entry in `EXCLUDED_DIR_NAMES` (see above), since this app's own data directory lives under `%APPDATA%\com.carlo.schlag`. Since a missed event can't be fixed at the `notify` layer, `prune_stale_entries()` self-heals instead: once at startup (before the scan, off the `Arc<Mutex<Connection>>` shared with everything else), it stat-checks every already-indexed path in parallel via `rayon` and deletes rows for files that no longer exist (or that now match an exclusion rule, per the note above â€” that check runs first, since it's a cheap string comparison with no I/O). This is a real, if imperfect, mitigation â€” a deleted file can still show up in search for as long as the app keeps running after a missed event, but it will always be gone again after the next restart, instead of the row persisting forever. A full continuous reconciliation (polling this while the app runs, not just at startup) wasn't built â€” the startup pass was judged sufficient for now; revisit if missed events turn out to be common in practice.

**Search (`search.rs`)**: `search_files` is a Tauri command that queries the same `index.db` the Phase 2 indexer writes to, but through its **own** `Connection` â€” WAL mode (already on) explicitly supports concurrent readers alongside one writer, so this is the minimal-diff way to add the first reader rather than routing queries through the indexer's writer thread. Opened once in `lib.rs`'s `setup()`, wrapped in `Mutex<Connection>` (needed since `Connection` isn't `Sync`), `.manage()`d as Tauri state. Filename substring matching goes through an FTS5 virtual table (`files_fts`, `tokenize='trigram'`, external-content on `files`/`id`) rather than `files.name LIKE '%x%'` directly â€” a plain B-tree index can't accelerate a leading-wildcard `LIKE`, and a real substring-plus-filter query against the live ~1.5M-row index measured 4â€“7+ seconds before this, which is a real, felt slowness, not a theoretical one. `files_fts` is kept in sync via `AFTER INSERT/UPDATE/DELETE` triggers on `files` (created in `database::open()`, cheap regardless of table size), so ongoing indexing (including Phase 2's live `notify`-driven updates) keeps it current for free. Extension/size/date/folder/regex filters are plain `AND` clauses against `files.*` in the same query, dynamically appended only for filters that are actually set. Query strings under 3 characters fall back to a plain `files.name LIKE` scan â€” the trigram tokenizer can't extract a trigram from anything shorter, and such broad queries hit the result cap fast enough anyway (measured ~1ms) that the fallback's full-scan cost doesn't matter in practice.

**Filename search has a second matching mode, "keywords" (`build_keyword_match`), alongside the default "phrase" mode (`build_phrase_match`) above** â€” both are branches of `build_query(query, filters, keyword_mode)`, selected by a `keyword_mode: bool` parameter threaded through `run_query`/`search_files` from the frontend. Phrase mode (the pre-existing behavior) requires the whole query to appear as one contiguous run of characters; keyword mode requires every whitespace-separated word to appear *somewhere* in the name, in any order, but rejects a name containing only some of the words â€” explicit user request ("scatter the search value, but the items SHOULD contain those keywords... do not show a file if it contains like tester 123 if a file contains only tester man test"). `build_keyword_match` partitions the query's words by the same 3-character trigram threshold `build_phrase_match` already used: words â‰Ą3 chars each become their own quoted `files_fts MATCH` phrase (all ANDed together into one combined MATCH string, since FTS5 only allows one `MATCH` per table per query), words <3 chars each become their own `files.name LIKE '%word%' ESCAPE '\'` clause ANDed in alongside â€” covered by `keyword_mode_handles_a_mix_of_short_and_long_words`, which mixes a 2-char and a 6+-char word in the same query. Filter-appending (extension/size/date/folder/regex) is unchanged and shared by both builders.

**The `folder` filter matches a trailing path separator, not just a raw prefix** (`path LIKE '{folder}\%'`, not `path LIKE '{folder}%'`). A plain prefix match also matches the folder's own row â€” any string trivially "starts with itself" â€” which surfaced as a real, confusing bug once folder-scoped search (below) shipped: search from inside a folder for a query matching that folder's own name (e.g. standing in `Documents`, searching "Documents") returned the folder itself as a "result," since its `name` matched the query *and* its `path` trivially satisfied the scope filter. The fix appends a separator before the wildcard (escaped correctly for `ESCAPE '\'` â€” the whole `folder + "\\"` string goes through `escape_like` together, not the folder alone with a raw `\` tacked on after, which would misparse as an escape sequence for the following `%`) so the filter only matches *contents* of the folder, never the folder entry itself.

**Content search (`content_index.rs`)**: a second, independent index â€” a Tantivy full-text index at `{app_data_dir}/content_index/`, separate from the SQLite `files`/`files_fts` filename index above â€” over what's *inside* extractable files, checked case-insensitively via `is_extractable()` against `EXTRACTABLE_EXTENSIONS`: `.txt`/`.md`/`.markdown`, `.pdf`/`.docx`/`.xlsx`/`.pptx`, and a broad set of code/config extensions (`.js`/`.ts`/`.py`/`.rs`/`.go`/`.java`/`.c`/`.cpp`/`.cs`/`.rb`/`.php`/`.sh`/`.ps1`/`.sql`/`.html`/`.css`/`.json`/`.yaml`/`.toml`/`.xml`/`.vue`/`.svelte`, etc.) â€” added on request once it was clear these are just plain text, no different from `.txt`/`.md`, so `extract_text()`'s dispatch match has a single `_ => extract_plain_text(path)` fallback arm covering all of them rather than one match arm per extension. **Legacy binary Office formats (`.doc`/`.xls`/`.ppt`) are a deliberate exclusion, not an oversight** â€” they're the old OLE2/Compound File Binary container, structurally nothing like the "zip of XML parts" the DOCX/XLSX/PPTX extraction below depends on; supporting them would need a real binary-format parser, not a longer whitelist, and they're rare in practice since Office defaulted to the modern formats back in 2007. `extract_text()` dispatches by extension: `String::from_utf8_lossy` over raw bytes for the plain-text fallback arm (never `read_to_string`, which would hard-fail and skip an otherwise-fine file over one stray non-UTF8 byte); `pdf-extract` for PDF; `calamine` for XLSX (iterates every sheet's used cells); and one shared zip+XML helper for **both** DOCX and PPTX, since both formats are "a zip of XML parts" (OOXML) and both put visible text runs in an XML element with local name `t` regardless of namespace prefix (`w:t` in DOCX, `a:t` in PPTX) â€” this was a deliberate choice over higher-level crates like `docx-rs` or the several surveyed pptx-specific crates (`pptx-to-md`, `rust-pptx`, `msoffice_pptx`), which are lower-maturity/smaller-community than `zip`+`quick-xml` (which most of them use internally anyway) â€” one low-level dependency pair covers two formats instead of two higher-level ones. PPTX slides are read via `ppt/slides/slideN.xml`, sorted **numerically** by N (`slide_number()`), not lexicographically â€” a plain string sort would put `slide10.xml` before `slide2.xml`. A source file over ~20MB is skipped outright, and extracted text over ~500,000 chars is truncated, not rejected â€” both are `ponytail:`-documented ceilings bounding pathological-file hang risk and per-doc Tantivy storage growth.

**Content indexing runs on its own dedicated thread/channel (`indexer.rs`'s `ContentEvent`/`run_content_indexer`), independent of the metadata scan**, because content extraction (parsing a PDF/DOCX/etc.) is far slower per-file than the metadata scan's stat calls â€” inlining it would slow down the already-fast filename index. `scan_drive`/`walk_subtree`/`apply_event` all additionally queue a `ContentEvent::Index(path, modified_ms)`/`Remove(path)` (via `queue_content_event`/`queue_content_removal`) alongside their existing SQLite work, for any extractable path. One dedicated consumer thread owns the Tantivy `IndexWriter` exclusively and drains that channel â€” no `Mutex<IndexWriter>` needed (unlike `Mutex<Connection>`), since a single-consumer-via-channel is the natural fit for a resource that isn't safely shared across threads for concurrent writes anyway. Commits are batched (every 50 docs or 5 seconds, whichever comes first), same batching rationale as the metadata scan's `flush_batch`.

**Re-extraction is skipped when a file's content hasn't changed, via a new `content_index_state(path, indexed_mtime)` SQLite table** (`database.rs`, created the same `CREATE TABLE IF NOT EXISTS` way as everything else â€” no `ALTER TABLE` migration needed). Phase 2's metadata scan already re-walks every drive on **every app launch** (upserts are idempotent, so this was always fine for cheap metadata) â€” but content extraction is expensive enough that redoing it unconditionally on every launch would be a real, felt slowdown for anyone with a real Documents folder full of PDFs/Office docs. Before extracting, `run_content_indexer` checks this table; a path is only re-extracted if its current `modified_ms` no longer matches what's recorded. This also means **no separate backfill step was needed** for the Tantivy index's first-ever build: the table starts empty, so the very next full scan naturally treats every extractable file as new â€” exactly the backfill behavior wanted, for free, mirroring how `files_fts`'s own triggers need no backfill on a brand new install.

**`search_content` exposes snippets as plain `fragment` text + `highlight_ranges`, never pre-built HTML.** Tantivy's own `Snippet::to_html()` wraps matched terms in `<b>` but does **not** HTML-escape the surrounding fragment â€” and that fragment is raw *file content* the app has no control over, so rendering it as trusted HTML would be a real XSS vector (a file whose text happens to contain `<script>`). The frontend builds highlighted spans itself via plain (auto-escaping) JSX string-splitting instead (`lib/highlightSnippet.ts`'s `splitHighlights`) â€” `dangerouslySetInnerHTML` is not used anywhere in this feature.

**Tantivy's `Snippet::highlighted()` ranges are *byte* offsets into the UTF-8 fragment â€” but the frontend is JavaScript, which indexes strings in UTF-16 code units.** These coincide for ASCII text but diverge for anything else (accented characters, non-Latin scripts, emoji), which would silently mis-highlight or crash on a non-character-boundary slice. `content_index.rs`'s `byte_ranges_to_utf16()` converts before the result ever reaches the frontend (building a byteâ†’UTF-16 offset map via `char_indices()`), so the frontend can just slice its string directly with no encoding awareness of its own â€” covered by a unit test using a multibyte (`Ă©`) fixture, not just ASCII.

**Folder scoping for content search is a post-filter over an overfetched candidate window, not a native Tantivy query** â€” `search_content` asks Tantivy for `RESULT_LIMIT * OVERFETCH_FACTOR` (4Ă—) candidates when a folder filter is set, then filters by path prefix in Rust before truncating to the real limit. `ponytail:`-documented ceiling: if a folder-scoped query needs more than that overfetch window to find enough in-folder matches, results may be incomplete; revisit with a real Tantivy-side path-prefix query if that proves common in practice. Content search's own metadata (`name`/`is_dir`/`size`/`modified_ms`) is looked up from the existing SQLite `files` table per surviving candidate, not duplicated into Tantivy's schema â€” Tantivy only stores `path`/`content`/`modified_ms`, the latter unused by search itself (the `content_index_state` table above is the actual source of truth for the re-extraction-skip decision).

**Content search's own core query logic lives in a plain `run_content_query()` function, separate from the `#[tauri::command] search_content` wrapper** â€” mirrors `search.rs`'s own `build_query`/`run_query` split from its `search_files` command, and for the same reason: the actual logic (query parsing, folder-prefix filtering, SQLite metadata join, snippet generation) is directly unit-testable without needing to construct Tauri `State` wrappers.

**Content search wraps every query in a quoted phrase (`phrase_query()`) before handing it to Tantivy's `QueryParser`, rather than parsing the raw string.** Tantivy's default parser splits unquoted input on whitespace and scores each word independently (any-of-these-terms), so an unquoted search for "am a file" would match any document containing "am", "a", or "file" *anywhere*, not that run of words together â€” a real, reported UX bug ("the search value shouldn't break") once content search was exposed to a general search box rather than a narrow "find this exact term" field. Filename search never had this problem: its own FTS5 phrase (`search.rs`'s `fts_phrase()`) already wraps the whole query as one quoted MATCH, since the trigram tokenizer's substring-search design made that the natural choice from the start. Wrapping unconditionally (not just for multi-word input) is deliberate â€” a single-word query wrapped in quotes parses identically to the same word unquoted, and this is meant to behave like "find this text," not expose a query language with AND/OR/field:value syntax. Covered by a test that indexes one document containing the phrase verbatim and a second containing the same three words scattered out of order, asserting only the first matches.

**Content search gets the same phrase-vs-keywords split as filename search, via `keyword_query()` next to the existing `phrase_query()`.** `run_content_query`/`search_content` take the same `keyword_mode: bool` parameter as `search.rs`'s equivalents, and build the query text passed to Tantivy's `QueryParser` accordingly: `phrase_query(query)` (unchanged) for the default, or `keyword_query(query)` for keyword mode. `keyword_query` deliberately does **not** use Tantivy's `set_conjunction_by_default()` API â€” it quotes each whitespace-separated word individually via the existing `phrase_query()` helper and joins them with a literal `" AND "` in the query text itself, so a raw word containing something that looks like Tantivy query syntax (a colon, a paren) still can't reach the parser unescaped, same reasoning already documented above for `phrase_query`. Covered by an extension to the existing phrase-mode test (`run_content_query_treats_the_query_as_a_contiguous_phrase_not_separate_words`): the same two fixture documents (one containing "am a file" verbatim, one containing the same three words scattered) both match in keyword mode, while a query substituting one word for one that's absent from both matches neither.

**Preview (`preview.rs`)**: deliberately the smallest possible new backend surface, because almost every format category needed **zero** new Rust logic. `app.security.assetProtocol` (`tauri.conf.json`, `{ "enable": true, "scope": ["**"] }` â€” broad scope for the same reason `opener:allow-open-path` already uses `**`: files worth previewing can be on any drive, not some app-scoped folder) is what lets the frontend hand the webview a direct file URL (`convertFileSrc()`) for Images/Video/PDF â€” the webview decodes/renders these natively (`<img>`, `<video controls>`, and `<embed type="application/pdf">`, the last relying on WebView2's own built-in Chromium PDF viewer for real paginated rendering, not a bundled library). **Enabling `assetProtocol` requires a full `npm run tauri dev` restart**, same as any other `tauri.conf.json` change â€” Tauri compiles capability/config into the binary at build time. Markdown/Text/Office preview reuses `content_index::extract_text()` outright â€” that function was already `pub` and already had its own per-type dispatch and size ceilings from Phase 3's content-search work, so `preview_text` is a one-line wrapper with no extraction logic of its own; it's the frontend's `previewKind()` categorization (`lib/previewKind.ts`) that ensures this command is only ever called for markdown/text/office paths â€” PDF's own `extract_text` branch (used for content search's indexing, not preview) is simply never reached from here. **Archive preview (ZIP only) is the one genuinely new piece of backend logic** â€” `read_archive_entries()` (split from the `list_archive_entries` command wrapper, mirroring `search.rs`'s `build_query`/`run_query` split, for the same directly-unit-testable reason) opens the file via `zip::ZipArchive` (already a dependency, from the DOCX/PPTX extraction work) and returns a flat `(Vec<ArchiveEntry>, bool)` â€” name/size/is_dir per entry plus a `truncated` flag, capped at a `ponytail:`-documented 2000 entries. Non-ZIP archive formats (rar/7z/tar.gz) aren't supported â€” `zip` only reads ZIP.

**Office preview shows extracted plain text, not the original document layout** â€” a small banner in `PreviewPane.tsx` says so explicitly ("Showing extracted text, not the original formatting") rather than silently shipping something that looks like it's trying and failing to be a real DOCX/XLSX/PPTX renderer. Real WYSIWYG rendering (fonts, tables, slide layout) would need a much heavier renderer per format; reusing the existing text-extraction pipeline was the deliberate cheapest-correct choice for this pass, upgradeable later if that proves insufficient.

**Markdown rendering uses `react-markdown`, not a markdown-to-HTML-string library plus `dangerouslySetInnerHTML`.** Same XSS reasoning already documented for content-search snippets: the markdown being rendered is arbitrary file content the app doesn't control, so trusting it as raw HTML would be a real vector. `react-markdown` parses to an AST and renders through actual React elements â€” raw HTML embedded in a markdown file is neither executed nor unescaped-inserted by default (no `rehype-raw` plugin is used), so there's no `dangerouslySetInnerHTML` anywhere in this feature either. Its handful of rendered elements (headings/paragraphs/lists/code/blockquotes/links) are hand-styled via a `.markdown-preview` CSS block in `App.css` rather than pulling in `@tailwindcss/typography` as a dependency â€” that plugin's generic defaults wouldn't match this app's specific density/type scale, and only a handful of elements ever need styling here.

**Window chrome (Phase 5): the native OS title bar is removed (`tauri.conf.json`'s window `decorations: false`), and the app draws its own.** The `TabBar` doubles as the title bar â€” a `data-tauri-drag-region` spacer moves the window (and double-click-maximizes), `WindowControls` provides minimize/maximize/close, and `WindowResizeHandles` re-adds edge/corner resizing (borderless Windows windows lose native resize borders). All window operations go through `@tauri-apps/api/window`'s `getCurrentWindow()` (frontend), gated by the `core:window:allow-*` permissions added to `capabilities/default.json` (`minimize`/`maximize`/`unmaximize`/`toggle-maximize`/`is-maximized`/`close`/`start-dragging`/`start-resize-dragging`/`internal-toggle-maximize` â€” the last is what makes `data-tauri-drag-region`'s double-click-maximize work). A `minWidth`/`minHeight` (640Ă—480) was added so the now-freely-resizable window can't be dragged down to nothing. **Like every `tauri.conf.json`/capabilities change, this needs a full `npm run tauri dev` restart** â€” a bad permission identifier fails the Rust build outright (they're validated against `gen/schemas/desktop-schema.json` at compile time), and HMR never picks up config/capability changes.

## Non-obvious gotchas (cost real debugging time â€” don't rediscover these)

- **`useFileExplorer()`'s startup `init()` (loads quick-access dirs/drives, navigates to This PC) must be idempotent, not just "conventionally called once."** It was written assuming exactly one caller (`FileExplorerView`), until `SearchModal` also started calling `useFileExplorer()` (to reach `openFileLocation`/`renameEntry`/etc. for its context menu) â€” both mount at app startup, so `init()` now fires from two places. `file-explorer.store.ts` gained an `initialized` flag set synchronously (before the first `await`) at the top of `init()`, so a second call arriving before the first one's async work even resolves still sees it and bails, instead of double-fetching `quickAccessDirs`/`listDrives` and double-navigating to `THIS_PC`. Any future hook that also needs `useFileExplorer()`'s action functions hits this for free â€” the guard lives in the store, not in the call site.
- **`tauri.conf.json` â†’ `app.windows[].dragDropEnabled` must be `false`.** Tauri's window has its own native OS drag-drop hook (for detecting files dragged in *from* the OS), on by default, and it intercepts native drag events at the WebView2/OS layer *before* they reach the page's own HTML5 `dragstart`/`dragover`/`drop` â€” including drags that originate entirely inside the webview. Without this set to `false`, in-app row dragging silently doesn't work.
- **Tauri plugin permissions often need an explicit scope, separate from the permission bit.** `opener:allow-open-path` alone does *not* allow opening files â€” `open_path`'s Rust implementation checks a scope (allowed paths) independently of the permission, and an empty scope always denies. Capability entries needing a scope use the object form, not a bare string: `{"identifier": "opener:allow-open-path", "allow": [{"path": "**"}]}`. See `src-tauri/capabilities/default.json`.
- **`useDropTarget`'s `onDragOver` must set `e.dataTransfer.dropEffect` explicitly.** `preventDefault()` alone permits the drop logically, but the cursor still shows "no-drop" unless `dropEffect` is set to match (`"move"` or `"copy"`).
- **Dropdowns/menus positioned `absolute` inside an `overflow-hidden` ancestor get visually clipped**, even though they're in the DOM. `ContextMenu` and the breadcrumb ellipsis dropdown both use `position: fixed` anchored via `getBoundingClientRect()` instead. **That alone isn't enough** â€” anchoring to the raw click/button coordinates with no regard for viewport edges clips the menu when it opens near the bottom or right edge (confirmed live: `ContextMenu` did exactly this). The fix (`ContextMenu.tsx`) is measure-then-correct: render at the raw anchor position, then in `useLayoutEffect` (not `useEffect` â€” this must happen before paint or the correction visibly jumps) measure the element's actual rendered size via a ref and clamp `top`/`left` against `window.innerWidth`/`innerHeight`. `SearchModal`'s filters section sidesteps this class of bug entirely rather than needing the same fix â€” it's a plain inline disclosure in normal document flow inside an already-centered dialog, not a separately-positioned floating popover, so there's no anchor point to clamp in the first place. The breadcrumb ellipsis dropdown doesn't do the clamp fix yet (lower risk in practice since it's anchored inside the address bar, not at a window edge, but the same class of bug could still hit it with a long enough hidden-segment list).
- **A Tauri command's own parameter names (as opposed to a struct's serde fields) are matched camelCase by default, even when the Rust side is snake_case.** `search_files`/`search_content`'s new `keyword_mode: bool` parameter, invoked from JS as `{ query, filters, keyword_mode }`, failed at runtime with `invalid args \`keywordMode\` for command \`search_files\`: command search_files missing required key keywordMode` â€” Tauri's IPC argument matcher camelCases each `#[tauri::command]` parameter name unless the command opts out via `#[tauri::command(rename_all = "snake_case")]`, which none of this app's commands do. This is a *different* mechanism from `SearchFilters`'s own struct fields (`min_size`, `modified_after_ms`, etc.), which stay snake_case verbatim on both sides via plain serde with no `rename_all` â€” those matched fine because struct deserialization isn't run through the same camelCasing. Fixed by sending `keywordMode` (not `keyword_mode`) as the invoke() object key in `file-explorer.service.ts`, while keeping the Rust-side parameter itself snake_case.
- **Draggable table rows need `select-none` on the whole `<tr>`, not just one cell.** Starting a drag gesture from a cell without it lets the browser begin a native text-selection drag instead of firing the row's own `dragstart`.
- **`copy_entry`/`move_entry` run every destination through `unique_destination()` before touching disk.** Without it, copying/moving into a folder that already has a same-named entry either silently overwrites it or (when source == destination, e.g. copying into the file's own folder) fails with a Windows sharing-violation error. It auto-appends `" (1)"`, `" (2)"`, etc., matching Explorer/Finder â€” this is unrelated to the Phase 5 "Duplicate Detection" feature (that's content-hash based, across the whole drive; this is filename-collision handling for a single operation).
- **The app root (`FileExplorerView.tsx`) sets `onContextMenu={(e) => e.preventDefault()}`** to suppress the native browser context menu everywhere by default. `EntryTable` rows override this locally with their own `preventDefault()` + custom `ContextMenu`. Any new area that wants a custom right-click menu needs to do the same; anywhere else, right-click silently does nothing (intentional, not a bug).
- **Prefer a documented Win32 API over an informal `rundll32`/PowerShell+COM shell-out for anything that needs to reliably open a real OS dialog.** Both were tried for `open_with_dialog`/`show_properties` and both silently no-op'd (clean exit code, zero output, no visible window) â€” no error to catch, indistinguishable from the feature doing nothing. `ShellExecuteExW` (see above) fixed it outright. If a future feature still needs to spawn a helper process that shows GUI, don't use `CREATE_NO_WINDOW` on it (it can suppress the resulting window's focus/foreground activation) â€” use the tool's own hidden-window flag if it has one (e.g. PowerShell's `-WindowStyle Hidden`, which only hides its own console).
- **A `LIKE` query against an FTS5 external-content table's column does not get accelerated by the trigram tokenizer** on the SQLite version this project's `rusqlite` bundles â€” it was tried first (`files_fts.name LIKE '%x%'`) and measured *slower* than just scanning `files.name` directly, while also (separately, and worse) returning wrong result counts for real substrings that definitely existed in the index. The correct, version-independent way to substring-search a trigram-tokenized FTS5 table is a **quoted phrase `MATCH`** (`files_fts MATCH '"chrome"'`) â€” this is what actually uses the trigram index and returns correct results. Confirmed by cross-checking `MATCH` result counts against a known-correct `files.name LIKE` baseline on the live index before trusting it.
- **A SQLite `REGEXP` (or any per-row scalar function) that compiles anything expensive must cache the compiled result across invocations**, not recompile it inside the function body. `create_scalar_function` is called once *per row*, and a query re-invokes it with the *same* pattern for every row in a single query â€” compiling a `regex::Regex` fresh each time turned one query into a 300+ second scan against the real ~1.5M-row index. A `thread_local!` cache of just the most-recently-compiled pattern fixed it (a query never interleaves two different patterns on the same thread), cutting the same query to ~1.2s.
- **An expensive one-time SQLite migration (e.g. `INSERT INTO fts_table(fts_table) VALUES('rebuild')` to backfill an FTS5 index for pre-existing rows) must not run inside a function called from Tauri's `setup()` hook.** `setup()` runs synchronously before the window appears â€” a 118-second rebuild measured against the real ~1.5M-row index would make every existing user's *next app launch* hang with no visible window and no explanation. `database::open()` only does cheap, size-independent schema/trigger creation; `database::backfill_fts_if_needed()` (the row-count-comparison + potentially slow rebuild) is a separate function, called only from the indexer's background thread. A brand new install never pays this cost at all â€” `files` is empty when the schema is first created, so the triggers alone keep `files_fts` current as rows stream in.
- **A number typed into an HTML `<input type="number">` is not actually bounded by anything unless you enforce it yourself.** `min`/`step` attributes are presentational hints, not runtime constraints on the JS value you read from `onChange`. `SearchFilters`' `min_size`/`max_size` are `u64` on the Rust side â€” a negative value typed into the size filter serializes fine as JSON but fails to *deserialize* into `u64` at the Tauri IPC boundary, rejecting the whole `invoke()` call and surfacing as a search error. `SearchFiltersFields.tsx`'s `parseNonNegative()` clamps and floors before it ever reaches `set()`. Found by testing the actual deserialization path (`serde_json::from_value` against a value shaped like what the frontend sends) rather than assuming an `Option<u64>` "just works" â€” the Rust-side unit tests never construct `SearchFilters` through JSON, so they can't catch this class of bug.
- **Every input that can trigger a search must be debounced consistently, not just the main query box.** `useSearch.ts` originally only debounced `search.query`, while `search.filters` (the extension/folder/regex text filters) went straight into the effect's dependency array â€” meaning every keystroke in a filter field fired an immediate, undebounced backend call, unlike typing in the search box itself. Both now go through `useDebouncedValue`.
- **A `notify` watcher must be registered *before* the work that could take a while, not after.** Registering it (and starting to drain its events) only once the entire multi-drive initial scan finishes meant a file created mid-scan fell into a real gap and was never indexed until the next full app restart happened to re-cover it â€” confirmed live by creating a file seconds after launch and finding it missing from search. The fix has two parts, both required: `start_watchers()` runs before scanning (queues events harmlessly if nothing's draining them yet â€” notify's OS-level watch and the channel itself don't need a reader to exist first), and `drain_events()` runs on its own thread *concurrently* with the scan (via a shared `Arc<Mutex<Connection>>`), not sequentially after it â€” otherwise a live create still wouldn't show up until the whole scan (minutes, for a real drive) completed, which reads as "still broken" even though nothing is technically lost anymore.
- **An exclusion check on a multi-path filesystem event (like a rename) must be applied per-path, not to the event as a whole.** `apply_event` used to skip the *entire* event if any of its paths were excluded â€” which broke deletion, since deleting a file moves it into `$RECYCLE.BIN` (excluded) and `notify` reports that as a rename (old real path â†’ new excluded path). Skipping the whole event meant the old, non-excluded path never got deleted from the index, so a deleted file kept showing up in search forever. Verified with a real Recycle Bin delete (`Shell.Application`'s `InvokeVerb("delete")` via PowerShell â€” the actual Explorer mechanism, not a plain `rm`, which just permanently removes the file and would've masked this bug entirely).
- **Vite silently base64-inlines any imported asset under its default 4kb threshold â€” including inside `import.meta.glob`.** `lib/fileTypeIcon.tsx`'s glob over all 1250 `material-icon-theme` SVGs looked harmless (most are just a few hundred bytes to a couple KB), but ~1225 of them landed under that threshold, so Vite embedded their actual image bytes as base64 text straight into the main JS bundle instead of leaving them as separate files â€” measured: a real production `vite build` came out to 1.7MB (366KB gzip) for the main chunk. The fix is the glob's `query: "?url&no-inline"` (not just `"?url"`, which still respects the inline threshold) â€” forces every matched SVG to stay a real, separate file that an `<img>` only loads when actually rendered on screen, instead of being parsed/evaluated as part of every app startup regardless of whether that file type ever appears. Confirmed via a before/after production build: main chunk dropped to 706KB (180KB gzip) after adding `&no-inline`, with the 1250 SVGs correctly showing up as separate `dist/assets/*.svg` files instead.
- **A `@tanstack/react-virtual` row whose real rendered height can differ from its `estimateSize` guess needs `measureElement` wired up â€” skipping it "because rows are uniform enough" is exactly the assumption that broke.** `EntryGrid.tsx` originally used a fixed constant (`tile + 32` per tile row, `HEADER_ROW_SIZE` per header row) with no `measureElement`/`ref`/`data-index` on the rendered rows, on the theory that real content always matches the estimate closely enough. It didn't, closely enough to matter: real, reported bug â€” the scrollbar thumb's length visibly shrank while scrolling down and grew back while scrolling up, reversibly, never settling. The instinct while debugging this was to suspect the *library's* internal bookkeeping (its measurement cache, its `getTotalSize()` computation) rather than the far more mundane explanation â€” our own estimate simply didn't match the real DOM, and with no measurement wired up, nothing ever corrected the mismatch. The fix is the textbook one: `measureElement: (el) => el.getBoundingClientRect().height` in the virtualizer options, plus `ref={virtualizer.measureElement}` and `data-index={virtualRow.index}` on each rendered row (and dropping the row's own explicit `height` style, letting it size itself naturally so there's something real to measure). Two other fixes were tried first and didn't hold up â€” moving the column-count `ResizeObserver` off the scrolling element onto a non-scrolling wrapper, and computing the wrapper's total height manually instead of trusting `getTotalSize()` â€” both reasonable-sounding, neither addressed the actual cause. Lesson: when a virtualized list's total size drifts, check whether the estimate matches reality (and wire up real measurement if not) before suspecting the library's internals.
- **`indexer.rs` test fixtures can't live under `std::env::temp_dir()` (or under the crate's own `target/`) once a name-based exclusion matches part of that path.** Adding `AppData` to `EXCLUDED_DIR_NAMES` (see the Indexing section above) broke five existing `apply_event`/`prune_stale_entries` tests outright â€” `std::env::temp_dir()` resolves to `%LOCALAPPDATA%\Temp` on Windows, and `path_is_excluded()` checks *every* path component, including ones the test itself never created, so any fixture built on it is silently pruned by the very code under test. The first fix attempt (nesting fixtures under the crate's own `target/` directory instead) broke the exact same way, just via a *different* list entry â€” `target` is itself excluded (Rust's own build-output pruning). The actual fix: a dedicated `test_scratch_dir()` helper pointing at `test-scratch/` â€” a sibling of `target/`, not a child of it, and not under `AppData` either â€” with its own `.gitignore` entry. Lesson: a test-fixture base path must be checked against the *same* exclusion list the code under test uses, not just against "does this look like a temp folder."
